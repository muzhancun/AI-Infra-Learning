# 第五章：内存层级、带宽、吞吐与延迟

> 本章把“内存层级”讲成一张成本地图，而不是一张硬件部件图。

我们前一章已经把 GPU 看成一台真正会“执行”的机器了。到了这一章，要再往前走一步：很多性能问题并不发生在“算”上，而是发生在“搬”上。一个 op 看起来只是在做一次矩阵乘、一次归一化，或者一次查表，但真正吃掉时间的，常常是数据从哪里来、要搬到哪里去、搬的时候能不能并行、以及搬完之后有没有被及时复用。

这就是为什么内存层级不能只当成结构图来背。更好的看法，是把它当成一张成本图。register 很快，但很少；shared memory 也快，但只在 SM 内部；L2 cache 比 HBM 更近，代价更低，但容量和命中都有限；global memory，也就是 GPU 直接挂着的那层大内存，才是大多数张量真正的家。张量如果一直待在最远的地方，计算就要反复把它拉近；如果能在更近的地方被重复利用，时间就会明显缩短。

> [!abstract] 直觉：这一章最重要的分界线
> 不是“我有没有这块内存”，而是“数据是不是已经在离计算更近的地方”。
> 同样大小的张量，放在不同层级里，代价会完全不同。

NVIDIA 的 CUDA Best Practices 直接把 shared memory 描述为 on-chip memory，并指出它通常比 local/global memory 具有更高带宽和更低延迟；而 NVIDIA 的 GPU Performance Background 则把 arithmetic intensity 与 `ops:byte` ratio 当成判断 math-limited / memory-limited 的第一性近似，并列出了 ReLU、LayerNorm 等典型 memory-limited 操作。([CUDA C++ Best Practices Guide](https://docs.nvidia.com/cuda/pdf/CUDA_C_Best_Practices_Guide.pdf), [GPU Performance Background User's Guide](https://docs.nvidia.com/deeplearning/performance/dl-performance-gpu-background/index.html))

## 从“能放下”到“搬得动”

初学系统的人容易先问容量：这个模型能不能放进显存？这个 batch 会不会爆？这些问题当然重要，但它们只是第一层。更细一点看，真正决定时间的，往往不是“能不能放下”，而是“每一步要搬多少次”。一个 tensor 如果被频繁读写，却没能在近处复用，它就会把时间消耗在数据移动上，而不是消耗在计算上。

这也是为什么 elementwise、reduction、normalization 这类操作经常看起来“很简单”，却未必快。它们做的数学工作并不多，但每个输出都要读进不少输入，读完还要写回去。计算量小，搬运量却不小，于是很容易落到 memory-bound：不是算得不够快，而是喂得不够快。

## 带宽、吞吐、延迟和并行度

带宽说的是单位时间里能搬多少数据，吞吐更像系统整体持续处理工作的能力，延迟则是一次搬运从发起到完成要等多久。并行度夹在中间：它决定这段时间里能不能同时干很多条搬运任务。把这四个词混在一起，容易看走眼；分开以后，很多现象就清楚了。

例如，一次访问很远的内存，单次延迟可能并不短，但如果后面能并发发起很多请求，硬件还能靠别的请求把空档填上。反过来，如果访问模式很散，数据一会儿这里一会儿那里，系统就算有带宽，也未必能把它吃满。于是性能并不是“带宽大就一定快”，而是“带宽、延迟、并行度和访问模式是否配合”。

看 trace 的时候，这个区分尤其重要。你会看到有些 kernel 并不算长，却在总时间里占了不小比例；这通常不是因为它计算复杂，而是因为它在等数据。也会看到有些看似很小的 op，反复出现在 timeline 上，像一串碎玻璃；它们单个都不重，但累计起来把很多时间都磨掉了。

## arithmetic intensity：先判断“算”和“搬”的比例

如果要用一个最朴素的指标来判断瓶颈方向，那就是 arithmetic intensity，也常被粗略理解成 ops:byte。它问的是：做了多少计算，搬了多少字节。这个比例高，通常说明计算更有机会成为主角；这个比例低，往往说明时间主要被数据移动吃掉了。

> [!abstract] 定义：arithmetic intensity
> 它描述“每搬一份数据，能做多少计算”。
> 比例高，算的成分更重；比例低，搬的成分更重。

> [!tip] 练习里为什么会写 matmul 的 FLOPs 约等于 `2 * m * n * k`
> 设矩阵乘法是 `A[m, k] × B[k, n] -> C[m, n]`。输出矩阵 `C` 一共有 `m * n` 个元素。`C` 里的每一个元素，都是一条长度为 `k` 的点积：要把 `A` 的一行和 `B` 的一列对应相乘，再把这些乘积加起来。
> 
> 如果用最朴素的计数方式看，一条长度为 `k` 的点积大致包含：
> - `k` 次乘法
> - `k - 1` 次加法
> 
> 所以更细一点的写法是 `m * n * (2k - 1)`；但在性能分析里，大家通常把它近似写成 `2 * m * n * k`，因为当 `k` 很大时，这个近似已经足够好，也更容易心算和比较量级。
> 
> 这里还有一个常见约定：如果硬件使用 FMA（fused multiply-add），一次 FMA 在性能统计里通常按 **2 FLOPs** 计算，也就是“一次乘法加一次加法”。所以你会经常看到 GEMM 的 FLOPs 写成 `2mnk`，这不是在说代码里一定显式写了两条指令，而是在用性能分析里最常见的 FLOPs 记账方式。

这个指标最有用的地方，不在于精确算出一个数字，而在于帮你快速分类。matmul 往往更像 compute-heavy 的工作，因为同样的数据会被重复使用很多次；softmax、layernorm、embedding lookup 则更容易露出 memory-bound 的一面，因为它们做的数学变换不算多，但数据进出很频繁。你不需要先写完整模型，只要先问：这个 op 的输出是不是需要把很多输入扫一遍？输入会不会被重复利用？中间结果会不会很快回到更近的层级？这些问题基本就能把方向判断出来。

> [!tip] 练习里为什么又把 matmul 的 bytes 近似写成“读 A 一次、读 B 一次、写 C 一次”
> 这是一个 **toy 级别的理想化记账**，目的是先抓住 arithmetic intensity 的大方向，而不是精确模拟真实 kernel 的每一次访存。
> 
> 在这个记法里：
> - `A` 的大小是 `m * k`
> - `B` 的大小是 `k * n`
> - `C` 的大小是 `m * n`
> 
> 所以如果每个元素占 `bytes_per_elem` 字节，那么最朴素的 bytes 估计就是：
> `((m * k) + (k * n) + (m * n)) * bytes_per_elem`
> 
> 这并不是说真实 GPU kernel 在物理上真的只读了一次 A、只读了一次 B。真实执行里会有 cache、tile、shared memory 搬运、重复访问、写回时机等很多细节。这里这样写，只是为了先回答一个更基础的问题：**如果数据复用做得还不错，matmul 的“算 / 搬”比例大概是什么气质？**

> [!tip] 练习里的 `numel` 是什么
> `numel` 是 **number of elements** 的缩写，意思是“张量里一共有多少个元素”。
> 
> 例如：
> - 一个向量形状是 `[1024]`，那么 `numel = 1024`
> - 一个矩阵形状是 `[1024, 4096]`，那么 `numel = 1024 * 4096`
> - 如果某个张量形状是 `[tokens, hidden]`，那么 `numel = tokens * hidden`
> 
> 在 toy 练习里它很常用，因为很多粗略估算都可以先从“每个元素要做几次事”开始。比如 ReLU 常常可以粗略记成“每个元素做一次简单运算”；LayerNorm 也可以先按“每个元素大概会参与几类运算、几次读写”来记账。  
> 所以 `numel` 不是字节数，也不是维度数，而是**总元素个数**。

roofline 模型可以看成是这个直觉的图形化表达：一边是算力上限，一边是带宽上限，中间把“算”和“搬”的关系放到一张图里。NVIDIA 的性能背景文档就是用 arithmetic intensity 这条轴，把 workload 分成 math-limited 和 memory-limited 两类。它很有帮助，但这一章不需要把它写成一门数学课。对我们来说，它的作用只是提醒：有些 op 还没碰到算力天花板，就已经先撞上内存墙了。([GPU Performance Background User's Guide](https://docs.nvidia.com/deeplearning/performance/dl-performance-gpu-background/index.html))

## 为什么 matmul 常常和 softmax、layernorm 不是一类问题

matmul 的形状和其他小算子很不同。它的好处，是同一批数据可以被重复利用很多次，所以只要调度得当，就有机会把更多时间花在计算上，而不是花在搬运上。也正因为这样，matmul 往往更能从更好的 tile、更好的并行和更好的数据驻留中获益。

softmax 和 layernorm 不是这样。它们通常要先扫一遍输入，再做一两个规约，最后再写回去。计算本身并不重，但读写路径却很完整。它们常常不是因为“太难算”而慢，而是因为“几乎每一步都得碰内存”。embedding lookup 更明显。查表的工作本来就不是算术密集型任务，它更像按索引把几块数据从远处搬过来，再拼到一起。所以这类 op 的速度，往往取决于访问是否连续、缓存是否命中、批次里相同条目是否能复用，而不是取决于乘法器有多强。

这也是为什么很多看似“简单”的层，反而最难优化。你如果只看算子名字，会觉得它们轻；你如果看数据路径，就会发现它们其实很重，只是重在搬运，不重在乘加。

## 回顾一下：LayerNorm 和 RMSNorm 到底在做什么

这一章里我们反复拿 LayerNorm 当 memory-bound 的典型例子，所以最好先把它的计算过程回忆清楚。否则很容易只记住“它慢”，却不知道它到底在忙什么。

先看 LayerNorm。假设输入张量的形状是 `[batch, seq, hidden]`，那么在 Transformer 里，LayerNorm 通常是**沿着最后一维 `hidden`** 做的。也就是说，对每个 token，都单独取出一条长度为 `hidden` 的向量，然后只在这条向量内部计算统计量，而不会跨 batch、也不会跨不同 token 混在一起算。

如果把某个 token 的 hidden 向量记成 $x = (x_1, x_2, ..., x_H)$，那 LayerNorm 大致做四件事。先算均值：

$$
\mu = \frac{1}{H}\sum_{i=1}^{H} x_i
$$

再算方差：

$$
\sigma^2 = \frac{1}{H}\sum_{i=1}^{H}(x_i - \mu)^2
$$

然后把每个元素做标准化：

$$
\hat{x}_i = \frac{x_i - \mu}{\sqrt{\sigma^2 + \epsilon}}
$$

最后再乘上可学习的缩放参数 $\gamma_i$，再加上可学习的偏移参数 $\beta_i$：

$$
y_i = \gamma_i \hat{x}_i + \beta_i
$$

这里的 $\gamma$ 和 $\beta$ 通常都是长度为 `hidden` 的参数向量。所有 token 共享同一组 $\gamma$ 和 $\beta$，但每个 token 的均值和方差是各自单独算的。你可以把它理解成：**每个 token 先把自己的 hidden 向量重新摆正，再交给后续层继续处理。**

> [!abstract] 定义：LayerNorm
> LayerNorm 是沿着单个样本的特征维度做归一化的操作。  
> 在 Transformer 里，它通常对每个 token 的 hidden 向量单独计算 mean 和 variance，再做 normalize、scale 和 bias。

从性能角度看，LayerNorm 的关键不在“数学特别复杂”，而在“它必须把整条向量很完整地碰一遍”。为了得到 mean，要读一遍；为了得到 variance，要继续处理这一整条向量；为了输出结果，还要做 normalize、乘 `gamma`、加 `beta`，最后再写回去。真实 kernel 当然可能会做 fusion、缓存和更细的优化，但从大方向上说，LayerNorm 的痛点常常不是乘加太难，而是数据流动太完整。这也就是为什么它经常更像 memory-bound，而不像 matmul 那样更容易接近 compute-bound。

RMSNorm 可以看成 LayerNorm 的一个更轻版本。它不再显式减去均值，而是只关心向量的整体尺度。它先算均方根，也就是 root mean square：

$$
\mathrm{RMS}(x) = \sqrt{\frac{1}{H}\sum_{i=1}^{H} x_i^2 + \epsilon}
$$

然后直接用这个量来做归一化：

$$
\hat{x}_i = \frac{x_i}{\mathrm{RMS}(x)}
$$

最后通常再乘一个可学习的 scale 参数：

$$
y_i = \gamma_i \hat{x}_i
$$

和 LayerNorm 相比，RMSNorm 少了一步“减去均值”的操作，也通常不需要对应的 $beta$。所以它更像是在回答这样一个问题：**这条向量整体有多大？我先把它的尺度调稳。** 这也是为什么很多现代 LLM 更偏爱 RMSNorm：它语义上更简单，计算和实现也往往更轻一些。

> [!abstract] 定义：RMSNorm
> RMSNorm 不做按均值居中的标准化，而是只根据向量的均方根大小来缩放。  
> 它通常比 LayerNorm 更简单，也常见于现代大语言模型。

不过从本章关心的角度看，LayerNorm 和 RMSNorm 都属于那种“数学不算特别厚、但需要把一整条向量认真读写一遍”的工作。也就是说，它们虽然细节不同，但在性能性格上常常还是更接近 memory-bound，而不是 compute-bound。

## 现实插页：AI researcher 真会这样 profile 一个 FLA layer

如果你在一个真实项目里想判断“这段工作到底是算太少还是搬太多”，很少会先手写 FLOPs 公式；更常见的做法，是先把一层真正跑起来，再看 profiler。`flash-linear-attention` 的 README 里就给出了直接实例化一个 token-mixing layer 的最小用法。把它稍微改一下，就能变成一个很典型的研究工作流：

```python
import torch
from fla.layers import MultiScaleRetention

device, dtype = "cuda", torch.bfloat16
hidden_size, num_heads = 1024, 4

# 这里直接实例化一个真实 layer；重点不是模型细节，而是让它真的跑到 GPU 上。
layer = MultiScaleRetention(
    hidden_size=hidden_size,
    num_heads=num_heads,
).to(device=device, dtype=dtype)

# 输入张量的 shape 是 [batch, seq, hidden]。
x = torch.randn(8, 2048, hidden_size, device=device, dtype=dtype)

# 先 warm-up，避免把 CUDA allocator 和首轮 lazy init 混进正式 profile
for _ in range(3):
    y, *_ = layer(x)

# torch.cuda.synchronize() 会让 host 等到当前 CUDA 工作真正做完。
# 因为很多 CUDA 调用默认是异步的，不 synchronize 就容易把前后两段时间混在一起。
torch.cuda.synchronize()

with torch.profiler.profile(
    activities=[
        # 同时记录 CPU 和 CUDA，才能看到 host 在喂活，device 在执行。
        torch.profiler.ProfilerActivity.CPU,
        torch.profiler.ProfilerActivity.CUDA,
    ],
    # record_shapes=True 会把输入 shape 也记下来，方便之后判断
    # 不同 shape 是否对应不同 op / kernel 行为。
    record_shapes=True,
) as prof:
    y, *_ = layer(x)
    # 让 profile 区间内发出去的 CUDA 工作都真正落完地，再结束采样。
    torch.cuda.synchronize()

# key_averages() 会把相同名字的事件聚合起来。
# sort_by="self_cuda_time_total" 表示先按事件自己消耗的 CUDA 时间排序。
print(prof.key_averages().table(sort_by="self_cuda_time_total", row_limit=20))

# 导出成 Chrome trace 后，可以在 chrome://tracing 或 Perfetto 里看时间线。
prof.export_chrome_trace("fla_eager_trace.json")
```

这段代码里最值得看的，不是模型名字，而是 trace 的形状。如果你看到少数几个厚实的 kernel 长时间占住 GPU，那往往更像 compute-heavy 的工作；如果你看到很多短 kernel 密密麻麻地排在一起，中间夹着 host 端空隙，那就更该怀疑 memory traffic、launch 开销或者算子过碎。换句话说，toy 里的 arithmetic intensity 和 roofline 不是要替代 profiler，而是帮你在看到真实 trace 时，更快地给这些形状起名字。

## trace 里会长什么样

把这一章和 profiling 连起来看，最直接的收获，就是你开始会用 trace 反推瓶颈。一个 memory-bound 的 kernel，通常会表现为计算片段短、读写占比高、单次执行看起来不复杂，但总时间却不小。一个更严重的情况，是很多 kernel 都很碎，每个都不长，却在 timeline 上密密麻麻地铺开；这时问题未必是某一个算子写得差，而是整个执行路径被切得太细，数据来回搬的次数太多。

所以，当你再看到一个慢 op，不要先急着问它是不是“算得慢”。先问三个更朴素的问题：它搬了多少？搬到哪里？搬完之后有没有被复用？如果这三个问题没有答清楚，讨论“优化算法”往往只是换了一种说法继续猜。

## 小结

这一章想留下的最重要的感觉，是把内存层级从“硬件列表”变成“成本地图”。register、shared memory、L2、global memory 并不是一张静态结构图里的几个名词，而是数据离计算远近不同所对应的不同代价。带宽、吞吐、延迟、并行度，也不是四个互相平行的概念，而是同一件事的不同侧面：数据能不能被及时搬到该去的地方。

如果只记住一句话，那就记住这句：很多模型问题看起来像算的问题，实际上先是搬的问题。你一旦开始用这个视角看 matmul、softmax、layernorm、embedding lookup，再看后面的 attention、KV cache 和多卡通信，系统的很多细节就会慢慢接上。

## 思考题答案

### 练习 1：arithmetic intensity toy

参考答案里，`matmul` 的 FLOPs/Byte 很高，而 `relu`、`layernorm`、`embedding` 的 FLOPs/Byte 都明显更低。这正好对应了本章最重要的直觉：**有些工作厚在计算上，有些工作厚在搬运上。**

**1）在这个 toy trace 里，哪些 op 最像“搬得多、算得少”，因此更值得先怀疑 memory-bound？**

最典型的是 `embedding`、`relu` 和 `layernorm`。其中 `embedding` 几乎可以看成“主要在搬数据”，因为它几乎没有多少算术工作；`relu` 也是每个元素只做一点点简单运算，但仍然要把输入读进来、把结果写回去；`layernorm` 虽然比 `relu` 多一些计算，但相对它要碰的数据量来说，算术密度还是不高。所以从 FLOPs/Byte 的角度看，它们都更像 memory-bound 的候选对象。相对而言，`matmul` 的 FLOPs/Byte 高得多，更有机会表现成 compute-heavy。

**2）如果 batch 变小、tile 变差，为什么 matmul 也可能没有原来那么“算力型”？**

因为 matmul 之所以常常像 compute-heavy，很大一部分前提是：同一份数据能被重复利用很多次，tile 组织也足够规整。一旦 batch 变小、矩阵形状变瘦、tile 不能很好铺开，数据复用就会下降，很多原本可以摊薄的搬运成本变得没那么容易摊薄。这样一来，matmul 的“有效 arithmetic intensity”就会变差，开始更明显地受到访存、启动开销和调度碎片的影响。也就是说，matmul 不是天生永远都偏算力，它也依赖问题规模和数据组织方式。

**3）为什么 embedding lookup 看起来几乎没有 FLOPs，却仍然可能在 profiler 里很慢？**

因为 embedding lookup 的难点本来就不在“算”，而在“取”。它更像按索引把若干行从远处拉过来，再写到输出里。只要访问模式不够连续、缓存命中不好、不同 token 之间缺乏复用，它就会把大量时间花在等待数据，而不是花在做乘加。也正因为这样，embedding lookup 是一个很好的提醒：**慢并不一定意味着算术复杂，也可能只是数据很难被高效搬到计算单元附近。**

### 练习 2：roofline toy

roofline 的作用，是把上一题的直觉再推进一步：不只是说“谁更像在搬”，还要问“在这台机器上，它更可能先撞上哪一面墙”。

**1）如果 trace 里某类 op 的 AI 很低，为什么它更容易先撞上带宽墙，而不是算力墙？**

因为 roofline 里 memory side 的上限大致可以写成：`arithmetic intensity × memory bandwidth`。当 AI 很低时，就算带宽本身不小，相乘之后的那条“内存上限线”也会很低。这意味着这类 op 还没来得及把计算单元喂满，就已经先被数据搬运能力卡住了。换句话说，每搬一字节只做很少计算的工作，天然更容易被带宽约束，而不是被峰值算力约束。

**2）为什么 matmul_like 更可能受益于更好的数据复用和更高的计算利用率？**

因为 matmul_like 的核心优势恰恰在于：同一块输入数据有机会参与很多次乘加。一旦 tile 设计更好、数据能更久地留在寄存器、shared memory 或 cache 里，这些数据就能在被再次搬运之前，先贡献更多计算。这样 arithmetic intensity 会更高，计算单元也更容易保持忙碌。所以 matmul_like 往往能从更好的复用、更规整的访存和更高的算力利用率中得到成倍收益。

**3）如果带宽翻倍但 peak TFLOPS 不变，哪些 op 最受益？这会怎样反映在 profiler 的时间占比里？**

最受益的会是那些本来就已经明显 memory-bound 的 op，比如 `embedding_like`、`relu_like`、`layernorm_like`。因为它们原先受限的就是带宽，带宽翻倍后，它们的 roofline 上限也会跟着抬高，执行时间有机会明显缩短。反过来，已经更接近 compute-bound 的 `matmul_like` 受益就会小很多，因为它原先卡住它的不是带宽，而是算力上限。

如果把这个变化放回 profiler 里看，常见的现象会是：那些 memory-bound kernel 的单次持续时间缩短，在 timeline 上占的时间比例下降；于是原本被它们淹没的一些 compute-heavy 工作，反而会在总时间里显得更突出。也就是说，优化带宽之后，瓶颈常常不会“消失”，而是会**转移**到下一面更紧的墙上。

## 本章 API 速记

- `torch.profiler.profile(...)`：打开一段 profile 区间，采集这段代码里的 CPU / CUDA 执行事件。
- `torch.profiler.ProfilerActivity.CPU`：告诉 profiler 记录 CPU 侧事件，比如 host 调用和 Python / runtime 开销。
- `torch.profiler.ProfilerActivity.CUDA`：告诉 profiler 记录 CUDA 侧事件，比如 kernel、memcpy 和 GPU 时间。
- `record_shapes=True`：把输入 shape 一起记下来，方便之后对照不同 shape 下的执行行为。
- `torch.cuda.synchronize()`：让 host 等到当前 CUDA 工作真正完成；常用来避免异步执行把计时或 profile 区间弄混。
- `prof.key_averages()`：把同名事件聚合起来，便于看某类 op / kernel 的总时间和平均时间。
- `table(sort_by="self_cuda_time_total")`：按事件自身消耗的 CUDA 时间排序，适合先看 GPU 上最重的事件。
- `prof.export_chrome_trace(...)`：把采集结果导出成 trace 文件，用 Chrome trace / Perfetto 一类工具看时间线。
