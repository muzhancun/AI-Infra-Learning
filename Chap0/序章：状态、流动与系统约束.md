
> 大模型系统并不是一堆 GPU 上跑着一个模型，而是一批状态对象在显存、网络和时间上的受限流动。

大模型系统之所以值得被当作独立研究对象，不是因为它用了更多 GPU，也不是因为它把工程规模放大了若干倍，而是因为当模型进入大规模训练、在线服务和 RL 后训练之后，决定成败的核心变量已经不再只是网络结构本身。很多时候，模型的数学形式几乎没有变，真正改变的是参数是否放得下、激活是否算得起、KV cache 能否复用、请求如何排队、checkpoint 会不会阻塞主路径、故障发生后多久能够恢复。于是，“系统”不再是模型外面的一层包装，而是模型得以存在、得以持续产生有用 token 的现实条件。

到 2026，这一点已经非常清楚。训练侧的主线收敛到 FSDP2、DCP、`torch.compile`、FlexAttention、Megatron Core；推理侧围绕 KV cache、prefix caching、continuous batching、disaggregated serving 展开；RL 后训练越来越像分布式数据流系统，关心的不只是优化目标是否正确，更是 rollout、过滤、学习和评估能否稳定重叠。AI Infra 研究的不是“层长什么样”，而是“状态在什么时间、驻留在哪、如何移动、由谁调度，以及这些选择怎样决定吞吐、延迟、稳定性和成本”。

这些名字第一次出现时很容易把人挡在门外，但它们背后的问题其实都很具体。FSDP2 可以粗略理解为“把训练状态切碎再按需拼回去”的分布式训练方式：参数、梯度和优化器状态不再完整地复制到每张卡上，而是沿数据并行维度分散存放，只在某一层真正要计算时临时聚齐。DCP 是 Distributed Checkpoint，它关心的不是模型怎么学，而是多机多卡环境下 checkpoint 如何并行保存、并行加载，以及当拓扑变化时如何重新切分。`torch.compile` 试图把 PyTorch eager 模式下零散的算子执行捕获成更大的计算图，交给编译器做融合、调度和图级优化；FlexAttention 则是近年出现的一类“既保留注意力形式灵活性，又尽量贴近高性能内核”的实现框架。Megatron Core 提供的不是某个单一算法，而是一套大规模训练积木，包括张量并行、流水并行、context parallelism、专家并行等。

推理侧的术语也最好先建立直觉。KV cache 指的是自回归生成时缓存历史 token 的 key/value 表示，这样模型在生成下一个 token 时就不必把整段上下文重算一遍。prefix caching 指多个请求如果共享同一段前缀提示词，就直接复用这段前缀已经算好的 KV。continuous batching 指 batch 不再是一开始就固定死的一组请求，而是可以在系统运行过程中动态进入和退出。disaggregated serving 则是把 prefill 和 decode 拆开：prefill 是把整段 prompt 跑一遍、把第一批 KV 建出来；decode 是在已有 KV 的基础上一个 token 一个 token 地继续生成。前者更像大吞吐矩阵计算，后者更像小步快跑的低延迟循环，所以它们未必应该绑在同一类资源上。

> [!abstract] 定义：本书里的 AI Infra
> 本书所说的 AI Infra，不是泛指“所有和 AI 有关的工程”，而是研究训练、推理和后训练中状态对象如何被放置、移动、调度与恢复的系统问题。
> 它最关心四类约束：显存、带宽、延迟、可靠性。
> 也正因为如此，它讨论的核心不再是层本身，而是状态流。

## 从模型对象到系统对象

如果只从模型视角出发，我们最熟悉的是层、张量形状、算子和 loss。它们当然重要，但还不够。系统视角关心的是另一组对象：参数、激活、优化器状态、KV cache、网络流量、调度以及时间。参数决定模型能否放下，激活决定长序列和大 batch 是否成立，优化器状态决定训练显存和 checkpoint 体积，KV cache 决定 serving 的内存经济学，网络流量决定多卡多机的扩展性，调度决定资源是否空转，而时间把这一切统一到 step、request、episode 三条轴线上。这里的 step 是一次训练更新，request 是一条在线推理请求，episode 则是在 RL 场景里从采样到回报的一段完整轨迹。系统工作之所以复杂，正是因为这三条时间轴上的“状态生命周期”完全不同。

这种转换并不抽象。训练时，一个 step 不是“做了一次前向和反向”这么简单，而是若干状态对象在显存、网络和存储之间完成了一次同步与重组；推理时，一个 request 也不只是“跑了一次解码”，而是一次 prefill、若干轮 decode、一些可能命中的 prefix，以及一份必须被管理和回收的 KV cache；在 RL 后训练里，一个 episode 更像一段跨组件、跨资源池的流水线，它的效率来自整个系统是否持续流动，而不是某一段代码局部有多快。只有把这些状态对象看清楚，AI Infra 才会从一组零散技巧变成一门系统科学。

> [!abstract] 定义：系统对象
> 在本章里，系统对象是那些必须被存放、搬运、同步、缓存或恢复的状态。
> 主要包括：参数、激活、优化器状态、KV cache、网络流量、调度、时间。
> 很多系统瓶颈，本质上都是这些对象的生命周期互相冲突。

## 训练时真正占内存和时间的东西

训练阶段最容易产生的错觉，是把成本都归因于“模型太大”或“算力不够”。实际上，一个训练 step 背后的账本远比参数量三个字复杂。它至少包括参数、梯度、优化器状态、激活、临时 buffer、通信 staging，以及 checkpoint I/O。这里的临时 buffer 指的是算子执行时需要的工作区，例如 GEMM 或 attention kernel 的中间 workspace；通信 staging 则是 all-gather、reduce-scatter 这类通信前后为了打包、拼接、暂存张量而额外占用的空间。真正决定能否扩展、能否稳定、能否持续跑下去的，往往不是某个单点峰值，而是这张账本在一个 step 内如何展开、如何回收、何处重叠、何处阻塞。

> [!abstract] 定义：训练 step 的状态账本
> 一次训练 step 不只是前向和反向。
> 它至少包含六类成本：参数、梯度、优化器状态、激活、临时 buffer/通信 staging、checkpoint I/O。
> 系统优化的关键，不是谁单独最大，而是谁在什么时候出现、持续多久、能否重叠或回收。

过去我们习惯把参数视为一个整体，把优化器状态视为训练附属物，把 checkpoint 视为训练完成后的保底动作。但现代训练栈早已不允许这种粗糙看法继续存在。FSDP2 的意义，不只是“更高效的并行实现”，而是它明确把参数、梯度和优化器状态当成可以分片、重组和按需展开的对象。对研究者来说，可以把它想成这样：传统数据并行是“每张卡都背一整本字典”，而 FSDP2 更像“每张卡只背自己负责的页，轮到某一页时大家再把那几页临时拼起来”。Megatron Core 的 context parallelism 进一步提醒我们，激活也不必天然绑定在单卡的完整序列维上，它同样可以被沿着序列切开，也就是把一条很长的上下文拆给多张卡分别持有，以换取长上下文下更可接受的显存和通信形态。`torch.compile` 与 FlexAttention 则把计算图和注意力路径变成可以被编译器与实现细节重写的对象，让“这段计算是否值得展开、是否会 graph break、是否能复用编译结果”成为训练效率的一部分。所谓 graph break，就是原本希望编译成一个整体图的执行路径在某处断开，又退回到零散的 eager 执行；一旦断得太多，编译收益就会被吞掉。

> [!abstract] 定义：FSDP2、DCP、graph break
> FSDP2：把参数、梯度和优化器状态按 rank 分片，计算时再按需聚齐。
> DCP：Distributed Checkpoint，多 rank 并行保存和加载训练状态，并支持重新切分。
> graph break：编译器本想捕获的一整段计算图在某处断开，重新退回 eager 执行。

DCP 则把另一件经常被低估的事情摆到桌面上：checkpoint 不是训练之后的附属品，而是训练路径中的 I/O 与恢复对象。checkpoint 可以理解为“把某一时刻的训练状态完整冻住”，其中往往不只包含模型参数，还包含优化器状态、随机数状态以及恢复训练所需的元信息。当规模上升到多机多卡，checkpoint save/load 以及失败后的 restart 开销，会直接进入系统的成本函数。异步保存真正解决的问题，也不是“写盘更优雅”，而是尽可能把 checkpoint 从关键路径挪开，让训练吞吐不必为恢复能力付出过高代价。于是，训练系统的中心问题不再是“能不能跑”，而是“能否在分片、通信、编译、I/O 和恢复共同约束下持续地产生有效 step”。

> [!tip] 直觉：为什么 checkpoint 是训练路径的一部分
> 只要训练会中断，checkpoint 就不是“最后才保存一下”的附属动作。
> 它决定了失败后多久能恢复、恢复时要搬多少数据，以及 save/load 会不会反过来拖慢训练主路径。

## 推理系统为什么围着 KV cache 转

如果说训练阶段的主角是参数与激活，那么推理阶段的主角往往就是 KV cache。训练的目标是让模型学到参数，推理的目标则是让请求在尽可能低的延迟和尽可能高的并发下持续生成 token。在这个过程中，参数固然重要，但它们往往是相对静态的；真正随负载形态、请求长度、路由决策和缓存命中率剧烈变化的，是 KV cache 的分配、复用、转移、淘汰与回收。

> [!abstract] 定义：KV cache
> KV cache 是自回归推理时为历史 token 缓存的 key/value 表示。
> 它的作用是避免每生成一个新 token 都把整段上下文从头重算。
> 它节省了算力，却把内存管理和缓存复用推到了系统中心。

这正是为什么现代 serving 系统的设计重心发生了迁移。像 vLLM 这样的系统，把 PagedAttention、continuous batching、automatic prefix caching、running/waiting requests、GPU cache usage、prompt tokens/s、generation tokens/s、prefix cache hit rate 直接变成日常监控对象。这里的 PagedAttention 可以理解为“把 KV cache 按页管理”，像操作系统管理虚拟内存那样把原本需要大块连续显存的缓存拆成较小块，从而减少碎片和浪费；automatic prefix caching 则是在系统层面自动发现可复用前缀，而不是要求用户手工管理。因为系统的吞吐并不是某个 request 单独跑多快决定的，而是由请求在队列里如何组合、prefill 和 decode 是否能够被统一调度、不同长度请求是否互相拖累、prefix 是否命中缓存共同决定的。推理系统一旦进入在线服务语境，排队策略本身就是性能的一部分。

> [!abstract] 定义：continuous batching、prefix caching、PagedAttention
> continuous batching：请求不是整批同时开始和结束，而是在运行中动态进入和退出同一个批次。
> prefix caching：共享前缀的请求直接复用已经算好的前缀 KV。
> PagedAttention：把 KV cache 按页组织，减少显存碎片和大块连续分配压力。

NVIDIA Dynamo 把这种逻辑推进得更彻底：服务被显式拆成 prefill engine 与 decode engine，前者负责生成 KV，后者接管并继续生成 token，系统围绕 TTFT、ITL、KV cache utilization、KV-aware routing 与 KV offloading 来组织资源。所谓 KV-aware routing，就是路由器在分发请求时不只看哪台机器空闲，还看哪台机器已经持有这条请求所需的 KV；所谓 KV offloading，则是当 GPU 显存不够时，把一部分不那么“热”的 KV 暂时转移到更慢但更大的存储层，例如 CPU 内存。SLA 是 service-level agreement，这里可以简单理解为服务对延迟、成功率和吞吐作出的外部承诺。这意味着推理系统的关键问题已经从“单机单批次如何做最快前向”转成“在分离式架构和动态负载下，KV 应该驻留在哪、何时移动、如何路由，才能在 SLA 约束下把更多请求稳定地撑住”。从这个角度看，推理并不是训练的简化版，而是另一套以缓存、排队和服务质量为核心的系统问题。

> [!tip] 直觉：为什么要拆 prefill 和 decode
> prefill 更像大吞吐的矩阵计算，decode 更像小步快跑的低延迟循环。
> 如果把两者强行绑在同一种资源和同一种调度策略上，通常两边都不会最优。

## AI Infra 关心的指标地图

系统研究如果没有指标，就只剩下口号；但如果指标选错了，又会把整个优化方向带偏。AI Infra 最重要的工作之一，就是为不同场景建立正确的指标地图。训练侧首先该看的不是笼统的“GPU utilization”，而是 MFU、step time、tokens/s、峰值显存、checkpoint save/load/restart 开销，以及 graph break 与编译命中情况。MFU 是 Model FLOPs Utilization，它问的不是“卡忙不忙”，而是“理论上能提供的矩阵乘法吞吐里，有多少真的变成了模型的有效计算”。step time 是一次参数更新从开始到结束花了多少时间，它通常比单个 kernel 的耗时更能暴露通信、同步和 I/O 问题。峰值显存与 checkpoint 开销则决定实验能否稳定、可恢复地持续推进。

> [!abstract] 定义：训练侧最核心的三个指标
> MFU：模型有效 FLOPs 占理论可用 FLOPs 的比例，更接近“算得值不值”。
> step time：一次参数更新的总时长，更接近“系统有没有卡在通信、同步或 I/O 上”。
> 峰值显存：某个时刻的最大 HBM 占用，决定配置能否真正跑起来。

推理侧的指标则必须贴着用户体验与服务约束来定义。TTFT 是 time to first token，也就是用户从发出请求到看到第一个 token 需要等多久；TPOT 往往指 time per output token，ITL 则是 inter-token latency，两者都在描述后续 token 之间的节奏是否稳定；E2E latency 是请求整体体验的总账。running/waiting requests 暴露的是排队与调度压力，queue interval 与 prefill interval 反映服务内部节奏是否失衡，prompt tokens/s 与 generation tokens/s 体现不同阶段的处理能力，GPU/KV cache utilization 与 prefix cache hit rate 则直接揭示缓存经济学是否成立。真正重要的问题不是某张卡看起来有多忙，而是在目标 SLA 下系统究竟能承受多大并发，并且能否稳定地维持住。

> [!abstract] 定义：推理侧延迟指标
> TTFT：从请求发出到第一个 token 返回的时间。
> ITL：相邻两个输出 token 之间的延迟。
> TPOT：平均每个输出 token 的时间，常与 ITL 一起使用。
> E2E latency：请求从开始到结束的总时长。

RL 后训练的指标更不该从 PPO 或 GRPO 的公式开始，而要从系统流动性开始。这里的 rollout 是用当前策略去采样一批轨迹，update 则是拿这些轨迹去做一次或多次参数更新。rollout 和 update 能否重叠，资源是否空转，队列是否堆积，版本同步是否及时，checkpoint 是否足够稳，这些指标决定的是系统是否具备持续学习能力。今天很多 RLHF 系统已经非常明确地朝这个方向演化：它们关心 hybrid engine、异步 trainer/rollouter、资源放置与 autoscaling。所谓 trainer/rollouter 分离，就是让一部分机器专门生成样本，另一部分机器专门学习；所谓 hybrid engine，则是在资源池层面更灵活地复用两者，尽量减少某一侧等另一侧时的 GPU idle。原因很简单，后训练的瓶颈越来越像数据流系统的瓶颈，而不是单机优化器的瓶颈。

> [!abstract] 定义：rollout、update、hybrid engine
> rollout：用当前策略生成样本或轨迹。
> update：用这些样本做一次或多次参数更新。
> hybrid engine：让采样和学习更灵活共享资源，尽量减少两侧互相等待造成的 GPU idle。

## 这章要反复纠正的三个误解

第一个误解是，`GPU utilization` 可以代表一切。它当然是有用信号，但远远不够。训练里更该先问 MFU 和 step time；服务里更该先看 TTFT、ITL 和并发下的退化曲线；RL 后训练里更该先看 rollout 与 update 是否真正重叠。很多系统表面上“卡很忙”，实际上忙的是等待、同步、排队和无效搬运。

第二个误解是，显存只是一个总数。事实上，显存从来不是抽象容量，而是不同状态对象争夺 HBM 的政治版图。HBM 可以简单理解为 GPU 上那块带宽很高、但容量昂贵且有限的主显存。参数、激活、优化器状态、KV cache 都在抢同一块物理资源，而它们的生命周期、访问模式和回收策略又截然不同。只说“这张卡有多少显存”几乎没有意义，真正的问题是：这块显存在什么时间被谁占用，占用多久，是否可以切分、转移、回收或外置。

第三个误解是，性能是 kernel 的局部属性。更快的 kernel 当然重要，但它只是系统中的一个局部片段。queue、prefill/decode 切分、cache hit、跨节点 KV 传输、checkpoint collective、版本同步、失败恢复，都会反过来决定一个系统整体看起来有多快。很多“模型问题”最后都不是通过再写一个更激进的 kernel 解决，而是通过重构调度、重排数据流、调整缓存与通信路径解决。AI Infra 的性能观，必须是全局的，而不是只盯着单点微优化。

## 让有用 token 连续地流过系统

把这些问题放在一起看，AI Infra 的核心其实可以收束成一句话：在显存、带宽、延迟和可靠性约束下，让有用 token 连续地流过系统。

训练里，我们用 MFU、step time 和 checkpoint 开销来衡量这种流动是否顺畅；推理里，我们用 TTFT、ITL/TPOT、cache hit rate 和 SLA 下最大并发来衡量它；RL 后训练里，我们用 rollout 与学习是否重叠、资源是否空转、恢复是否稳定来衡量它。不同场景看上去彼此不同，但它们都在回答同一个系统命题：状态对象如何在有限资源下被放置、移动、复用、回收和调度。

因此，这一章的目的不是替读者记住更多框架名，而是完成一次视角转换。只要这个转换发生了，后面的训练并行、推理 serving、KV cache、checkpoint、调度与后训练流水线，就不再是彼此割裂的工程细节，而会成为同一门系统科学在不同时间尺度上的展开。大模型系统的本质，不是把更多 GPU 堆在一起，而是让状态对象在受限世界里尽可能高效、稳定、可恢复地流动起来。
