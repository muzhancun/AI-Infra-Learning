
# 第 2 章：什么叫系统视角

## 本章定位

第一章已经回答了一个基础问题：AI Infra 研究的不是“模型结构”本身，而是参数、激活、优化器状态、KV cache、网络流量、调度与时间这些系统对象。第二章不再重复“对象是什么”，而是进一步回答另一个问题：**当这些对象进入一次真实执行时，它们如何构成一个系统过程。**

因此，本章的重点不是继续枚举对象，也不是提前展开 GPU、kernel、collective 或 profiling 的机制，而是建立一种统一的分析方法：把一次训练 step 或一次生成 request 看成若干种“流”在时间中展开、冲突、重叠和等待的过程。

这一章结束时，读者应当获得的不是更多术语，而是一种判断框架：以后看到训练并行、KV cache、checkpoint、continuous batching、rollout pipeline 这些具体技术时，能够先问一句——**现在到底是什么在流动，卡在了哪里，谁在等谁。**

---

## 2.1 系统视角不是换一套术语，而是换一个分析单位

这一节的任务，是把“系统视角”从抽象说法变成一个明确的方法起点。

从模型视角看，我们熟悉的分析单位是层、算子、loss、梯度和参数更新。模型被理解为一个函数，训练被理解为一个优化过程，推理被理解为一次前向计算。这种视角对理解模型本身是必要的，但一旦进入大规模训练、在线服务和 RL 后训练，它就不再足够。因为真正限制系统表现的，往往不是某一层“在算什么”，而是某些状态对象是否能放下、是否能及时移动、是否会阻塞其他步骤、是否能在正确的时间被回收或复用。

系统视角的变化，不在于换一批名词，而在于换一个分析单位。它不再把“层”当成最小观察对象，而是把“一次执行过程”当成最小观察对象。这里的执行过程，训练中体现为一次 step，服务中体现为一条 request 的生命周期，RL 中体现为一段 rollout 与 update 之间的闭环。对象还是第一章那些对象，但现在它们不再是静态账本，而开始具有时间结构：谁先出现，谁后出现，谁必须常驻，谁可以短暂存在，谁必须跨设备移动，谁在等待另一个状态准备好。

从这个意义上说，系统视角真正增加的内容，是“过程感”。同一个参数张量，在第一章里是一个系统对象；到了第二章，它就必须被放回时间中，去看它何时被取用、何时被同步、何时被重组。KV cache 也是一样。第一章里它是推理系统的核心状态，第二章里则要开始问：它在 request 的生命周期里如何建立、如何命中、如何延续、如何阻碍别的请求进入 batch。

这一节最后可以用一句话收住：

**第一章让我们看见系统里“有什么”，第二章要让我们看见这些对象“如何发生”。**

---

## 2.2 一次训练 step：把执行看成四种流

这一节是本章的第一块主体。它不追求硬件细节，而是追求“过程被拆开之后，读者第一次看见训练其实是一系列流的叠加”。

最自然的做法，是从一次训练 step 的最小链条出发：数据进入，形成 batch 或 microbatch；前向计算展开；激活被保留；反向传播回收并产生梯度；某些状态需要同步；最后参数与优化器状态更新，必要时还要落盘或等待下一步。这条链条在模型论文里常常被压缩成“forward + backward + update”，但在系统视角下，它其实是几类不同性质的流同时在推进。

首先是数据流。训练不是“数据来了就算”，而是数据被读取、切分、打包、分发，然后才进入计算。样本如何拼 batch，batch 如何拆成 microbatch，数据如何送入不同 rank，这些都属于训练系统的前半段组织问题。模型结构通常不改变这里的语义，但系统设计会改变这里的成本。

其次是内存流。参数、激活、梯度、优化器状态并不是一团静止的“占用”，而是在 step 的不同阶段以不同生命周期出现。参数要参与前向和反向，激活通常在前向生成、反向消费，梯度在反向中形成，优化器状态在更新时被访问。系统里真正难的地方，不是谁单独最大，而是这些对象的时间分布是否互相挤压。也就是说，训练的显存问题首先是“时序问题”，然后才是“总量问题”。

再往下是通信流。只要训练不在单卡上进行，某些对象就必须在设备间移动。某些中间状态需要交换，某些梯度需要同步，某些参数需要按需聚齐或重组。这里不需要在第二章展开 collective 的语义，但必须让读者第一次意识到：多卡训练不是把单卡训练平铺复制一遍，而是在原有前向/反向之外，多了一条独立的消息流。

最后是控制流。训练 step 之所以会“卡住”，往往不是因为某一段数值计算本身特别复杂，而是因为执行顺序里存在等待。什么能重叠，什么必须串行，哪里形成同步点，哪里会产生 bubble，哪里虽然 GPU 没空闲但系统仍然没有产生更多有效 step，这些都属于控制流问题。控制流真正关心的是执行关系，而不是数学关系。

这一节的结尾不需要再把四种流做术语复述，而要形成一个整体认识：

**一次训练 step 不是一次抽象的参数更新，而是数据流、内存流、通信流和控制流在一个时间窗口中的共同展开。**

这样，第二章和第一章就形成了明确分工：第一章已经说过训练时有哪些状态对象和成本；第二章现在补上的是，这些对象在一次 step 内到底如何按时间彼此纠缠。

## 2.3 一次生成 request：为什么 serving 不只是少了 backward

这一节与上一节构成镜像。它的目的，是防止读者把推理想成“训练删掉反向传播之后剩下的那一半”。系统上，这种理解是不够的。

一条 request 进入 serving 系统之后，它面临的不是一个简单的“前向计算”，而是一整段带状态的生命周期。它先进入队列，被系统接纳，与其他请求一起形成批次；然后在 prefill 阶段把已有 prompt 处理一遍，建立最初的上下文状态；之后进入 decode 阶段，在已有历史上一个 token 一个 token 地继续生成；直到输出结束，状态被回收，请求才真正离开系统。

这一过程一旦展开，四种流会再次出现，但它们的相对重要性已经和训练不一样了。

数据流在这里不再只是样本进入模型，而是 request 作为服务对象在系统中的推进。它会排队、等待、进入 batch、部分完成、继续保留在系统里。于是 serving 的数据流天然带有在线性和动态性。

内存流的中心也变了。训练阶段的核心张量是参数、激活、梯度和优化器状态；而生成阶段真正主导系统形态的，往往是 KV cache 这类历史状态。参数在推理时相对稳定，但请求长度、上下文长度、并发数量与缓存命中率，会持续改变 KV cache 的占用和布局。于是推理系统里，“内存问题”不再主要表现为训练式的状态账本，而表现为缓存如何建立、命中、复用、迁移和回收。

通信流在服务里也不是训练通信的缩小版。训练中的通信往往围绕梯度、参数切分和并行同步展开，而服务中的通信更容易被 request 动态性塑形。前缀是否已存在于某个设备上，请求是否要被转发到持有相关状态的资源，prefill 与 decode 是否在同一类资源上执行，这些都会改变通信模式。

控制流在 serving 中往往比训练更显眼。因为系统不只是要“执行一个图”，还要决定“谁先进入、和谁一起、等待多久、何时释放状态”。这时候，batching 策略、队列组织、prefill/decode 的切分方式，都会直接决定 latency 和 throughput 的形状。也正因为如此，serving 比训练更像一个带状态的调度系统。

这一节最后最重要的结论是：

**生成不是训练删去 backward 之后的残留物，而是另一种以状态延续和在线调度为中心的系统过程。**

这句话会把后面关于 KV cache、prefix reuse、continuous batching、disaggregated serving 的章节自然接出来，同时避免在第二章就提前讲太多实现细节。

## 2.4 从这一章开始，后面所有章节都在研究“流如何撞墙、如何被重写”

这一节是方法论收束，也是全书桥接。它不再引入新的对象，而是说明“四流分解”到底有什么用。

学会用数据流、内存流、通信流和控制流去看训练与生成之后，读者会发现，后面所有章节其实都在做两类事情。第一类，是解释这些流为什么会撞到某种资源边界。比如为什么某些训练配置最后变成显存问题，为什么某些服务配置最后变成 cache 管理问题，为什么某些多卡扩展最后变成网络与同步问题。第二类，是解释工程系统如何改写这些流。比如分片是在重写内存流，collective 和并行切分是在重写通信流，编译与 kernel 融合是在压缩控制流和内存流，serving 调度则是在重写 request 的控制流与状态驻留方式。

这样一来，后面的第一部“硬件、运行时与 profiling”就有了清晰位置。它不是另起炉灶讲 GPU 基础，而是在回答：当我们说某个流卡住了，它究竟卡在了什么硬件层级、运行时机制或调度路径上。再后面的训练并行、KV cache 与 serving、RL rollout 数据流，也不再是零散技巧，而都可以被看成“改写某种流”的系统设计。

本章因此应当以一句很短的总结收束：

**系统视角的价值，不是给模型加一层工程包装，而是把训练与生成重新理解为受资源约束的动态过程。**

## 本章写法建议

这一版最好不要再切太多二三级标题。整体保持四个主节即可，每节内部用连续自然段推进。这样有两个好处。

第一，它会明显减少与第一章的“定义型重复”。第一章已经承担了大量概念引入工作；第二章再大量切小标题，容易显得像在换一种格式重讲一遍。  
第二，它会让这一章更像“方法论展开”，而不是“术语手册”。第二章需要的是推进感，而不是目录感。

如果你后面要把它写成正式正文，建议也遵循这个节奏：少定义，多过程；少罗列，多对照；少框架名，多时间结构。
