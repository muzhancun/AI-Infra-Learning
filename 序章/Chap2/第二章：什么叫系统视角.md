> 系统视角的核心，不是多记几个组件名，而是把一次执行看成若干种状态流在时间中相遇、分离、等待与回收。

第一章已经把这本书最重要的一步迈出去了：我们不再把大模型只看成若干层和若干公式，而是开始把参数、激活、优化器状态、KV cache、网络流量、调度与时间当成真正的系统对象来观察。可这还只是第一步。只知道系统里“有什么”，并不等于已经理解了系统。真正让 AI Infra 成为独立研究对象的，不是对象清单本身，而是这些对象一旦进入真实执行，就会沿着时间展开、互相挤压、彼此等待，并最终共同决定吞吐、延迟、稳定性与成本。

所以第二章要做的，不是继续补一批定义，也不是急着进入 GPU、kernel、collective 或 profiler 的细节，而是先建立一种观察方法。以后无论你看到的是训练并行、KV cache、continuous batching、checkpoint，还是 RL rollout pipeline，都应该先退后一步，问同一个问题：现在到底是什么在流动，卡在了哪里，谁在等谁。只有这个问题先立住，后面的工程技术才会显得连贯；否则，一切都只会像零散技巧。

## 系统视角不是换一套术语，而是换一个分析单位

从模型视角看，我们最熟悉的分析单位是层、算子、loss、梯度和参数更新。模型被理解为一个函数，训练被理解为一次优化，推理被理解为一次前向。这种视角对理解模型为什么有效是必要的，但它默认了一件事：只要数学关系弄清楚，执行问题就只是附属成本。大模型系统恰恰打破了这个默认前提。到了多卡训练、在线 serving 和 RL 后训练阶段，决定系统表现的往往不再是哪一层“在算什么”，而是哪些状态必须常驻、哪些状态可以分片、哪些状态需要跨设备移动、哪些步骤能重叠、哪些地方会形成等待。

因此，系统视角真正变化的地方，不在于它换了一套更工程化的词，而在于它换了分析单位。它不再把“层”当成最小观察对象，而是把“一次执行过程”当成最小观察对象。训练里的这个过程是一整个 step，serving 里是单条 request 从进入到退出的生命周期，RL 里则是 rollout、筛选、打分、update 之间形成的闭环。对象还是第一章那些对象，但现在它们不再是静态账本，而开始带有时间结构：谁先出现，谁后出现，谁持续整个过程，谁只短暂占用资源，谁必须在另一个对象到位之后才能继续。

> [!abstract] 定义：系统视角里的分析单位
> 训练里，最小分析单位是一整个 step，而不是单层前向。
> serving 里，最小分析单位是一条 request 的完整生命周期，而不是某次局部 decode。
> RL 里，最小分析单位是 rollout 到 update 的闭环，而不是某个孤立的 loss。

这意味着，同一个对象放在不同章节里，含义会发生变化。参数在第一章里首先是系统状态，它决定模型能否被完整表示；到了这一章，它必须被重新放回时间中去看，去看它何时被取用、何时被分片、何时被同步、何时被重组。KV cache 也是一样。第一章里，它是 serving 的核心状态对象；第二章里，就要进一步问它在 request 生命周期里如何建立、如何延续、如何命中已有前缀、又如何阻碍其他请求进入批次。对象没有变，视角变了，于是问题也跟着变了。

系统视角真正增加的内容，可以概括成一句话：它增加了“过程感”。模型视角擅长回答“这个网络为何有效”；系统视角擅长回答“这个过程为何变慢、为何放不下、为何扩不起来、为何恢复不了”。两种视角不是替代关系，而是互补关系。没有模型视角，你不知道系统究竟在为哪种计算服务；没有系统视角，你就会高估抽象算法、低估现实执行，并不断把本来属于系统的问题误诊为模型问题。

所以第二章的任务不是宣告“从此只谈工程”，而是补上模型视角天然缺失的一半。第一章让我们看见系统里“有什么”，第二章要让我们看见这些对象“如何发生”。

## 一次训练 step：把执行看成四种流

模型论文常把训练压缩成一句极简的话：forward、backward、update。这个说法没有错，但它隐藏了系统里真正重要的东西。对于研究 AI Infra 的人来说，一次训练 step 不是一次抽象的参数更新，而是多种流在同一段时间里共同展开的结果。要把训练真正看成系统过程，最方便的方法是把它拆成四种彼此纠缠的流：数据流、内存流、通信流和控制流。

首先是数据流。样本并不是天然就等于 batch。它们先要被读取、切分、过滤、拼接，再被组织成 batch 或 microbatch，然后被送到不同 rank 上进入真正的计算。这个过程表面上看像训练前的预处理，实际上它已经进入 step 本身。样本如何分桶、长短序列如何对齐、microbatch 如何切分、输入如何分发，都会改变后续每一步的节奏。很多训练问题表面上像“GPU 算得不快”，本质却是数据还没及时到位，或者数据组织方式让后面的计算和通信更难重叠。

其次是内存流。参数、激活、梯度、优化器状态、临时 buffer 并不是一堆静止的“占用”，而是在 step 的不同阶段以不同生命周期出现。参数在前向和反向都要被访问，激活通常在前向产生、在反向消费，梯度在反向中逐步形成，优化器状态在 update 时被读取和修改。于是训练里的显存问题首先不是一个总量问题，而是一个时序问题。真正决定能否跑起来的，往往不是某个对象单独有多大，而是这些对象是否会在错误的时间重叠到一起，形成不可承受的峰值。激活检查点之所以有意义，也正是因为它在改写内存流的时间结构，用额外计算换更平缓的显存峰值。

> [!abstract] 定义：训练 step 的四种流
> 数据流：样本如何被读取、打包、切分并送入计算。
> 内存流：参数、激活、梯度、优化器状态等对象何时出现、持续多久、何时回收。
> 通信流：哪些对象必须跨设备交换、聚齐、同步或重分布。
> 控制流：执行顺序如何安排，哪里能重叠，哪里必须等待，哪里形成同步点和 bubble。

再往下是通信流。只要训练不在单卡内完成，系统里就天然存在一条独立于数值计算的消息路径。某些梯度要同步，某些参数要按需聚齐，某些激活或序列片段要在并行维度之间交换，某些优化器状态要在 save/load 或重分片时重新布局。这里第二章不需要提前讲清每一种 collective 的语义，但必须把一个判断立住：多卡训练不是把单卡训练复制多份，而是在原有前向与反向之外，又多出一条必须被设计、被调度、被隐藏延迟的通信流。如果看不见这条流，就很难解释为什么理论算力增长之后，实际 step time 并没有按比例下降。

最后是控制流。训练 step 之所以经常“卡住”，很多时候不是因为某一段矩阵乘法特别慢，而是因为执行关系里存在等待。哪些 microbatch 能先上，哪些通信能和计算重叠，哪些阶段必须等前一个 rank，哪些地方形成全局同步点，哪些空档虽然 GPU 没彻底闲着却没有产生更多有效 step，这些都属于控制流问题。所谓 bubble，本质上就是资源已经在线，却没有在那一小段时间里产生理想的有效工作。训练系统的成熟度，往往体现在能否把这些 bubble 压到足够小，而不只是能否把单个 kernel 写得更快。

四种流一旦放到同一张时间线上，训练就会从“参数更新”变成“过程协调”。你会开始发现，许多后面章节里看起来彼此无关的技术，其实都在改写这四种流的形状。分片是在重写内存流与通信流，编译和 kernel 融合是在压缩控制流中的调度开销，checkpoint 策略是在处理内存流与 I/O 流在更长时间尺度上的冲突。第二章不需要把这些技术细节提前讲完，但需要先把这张方法地图画出来。

因此，一次训练 step 不应再被理解成一个抽象公式，而应被理解成数据流、内存流、通信流和控制流在一个时间窗口中的共同展开。系统工作真正要优化的，从来不是其中某一个流的孤立极值，而是它们放在一起时是否还能持续地产生有效 step。

## 一次生成 request：为什么 serving 不只是少了 backward

如果训练系统容易被误解成“只是算得更大”，那么 serving 系统最常见的误解就是“它不过是训练删掉 backward 之后剩下的那半段”。从模型计算图上看，这种说法似乎成立；从系统过程上看，它基本是错的。因为一条 request 进入在线系统之后，面对的不是一次静态前向，而是一整段带状态、带排队、带调度、带缓存的生命周期。

一条 request 首先会进入队列，被系统接纳，与其他请求一起形成某种批次结构。然后它进入 prefill 阶段，把已有 prompt 整体编码一遍，建立最初的上下文状态；之后它进入 decode 阶段，在已有历史之上一个 token 一个 token 地继续生成；直到生成停止、状态被释放，这条 request 才算真正结束。只要把这个过程完整展开，serving 的系统性就会立刻显现出来。它不是单次前向，而是一个状态在时间中持续存在、不断被访问、不断被调度、最后再被回收的过程。

> [!abstract] 定义：prefill 与 decode
> prefill 是对已有 prompt 做整段前向，建立初始上下文状态。
> decode 是在已有历史基础上逐 token 继续生成，并持续复用已有状态。
> 两者都在“前向”这个大类里，但资源形态、调度目标和性能指标并不相同。

在这个过程中，四种流会再次出现，但它们的重要性排序已经和训练不同。数据流在这里不再只是“样本进入模型”，而是 request 本身在系统中的推进。请求会排队、被接纳、进入 batch、部分完成、继续留在系统里等待下一轮 decode。于是 serving 的数据流天然带有在线性和动态性。训练里的 batch 往往在 step 开始前就已经确定，serving 里的 batch 却是活的，它会随着请求的进入和退出不断变化。continuous batching 的价值，也正是把这种动态性当成系统常态来设计，而不是拿离线训练的静态批次想象去套在线服务。

内存流的中心在 serving 中也发生了转移。训练时最显眼的是参数、激活、梯度和优化器状态；生成时真正主导系统形态的往往是 KV cache 这类历史状态。参数在推理阶段相对稳定，但请求长度、上下文长度、并发数量、prefix 命中率会不断改变 KV cache 的占用和布局。于是 serving 里的“内存问题”不再主要表现为训练式的显存账本，而表现为缓存如何分配、如何延续、如何复用、如何迁移以及如何回收。很多系统设计看起来像调度技巧，本质上其实是在为 KV cache 的生命周期管理服务。

通信流在 serving 中也不是训练通信的缩小版。训练里的通信大多围绕参数、梯度和并行切分展开，而服务里的通信更容易被 request 的动态状态塑形。某个前缀已经缓存在什么地方，请求是否应该被路由到持有相关状态的设备，prefill 生成的 KV 是否要转交给 decode 资源池，某些长上下文状态是否需要跨层级迁移，这些都会改变消息路径。尤其当 serving 开始做 disaggregated 架构之后，通信就不再只是“算完再发一点数据”，而变成 request 生命周期的一部分。

控制流在 serving 里往往比训练更显眼。系统不只是要执行一个计算图，还要持续决定谁先进入、和谁一起、等待多久、何时释放状态、何时让更短或更紧急的请求穿过队列。batch window、队列策略、prefill 与 decode 的协调方式、prefix reuse 的命中与否，都会直接改变 TTFT、ITL 和整体并发能力。也正因为如此，serving 比训练更像一个带状态的调度系统。性能不再只是单次计算的局部属性，而是队列组织、缓存驻留和调度决策共同塑造的全局结果。

所以，生成并不是训练删去 backward 之后的残留物，而是另一种以状态延续和在线调度为中心的系统过程。只要这个判断一成立，后面关于 KV cache、prefix reuse、continuous batching、disaggregated serving 的章节就都有了统一入口。它们研究的不是若干孤立技巧，而是在一条 request 的生命周期里，如何重写内存流、通信流和控制流，让系统在延迟与吞吐之间找到更稳定的平衡。

## 从这一章开始，后面所有章节都在研究“流如何撞墙、如何被重写”

把训练 step 和生成 request 都放到四种流的框架里之后，这本书后面的结构其实会一下子变得很清楚。后面所有章节看上去在讨论不同主题，实际上都只是在做两类事情。第一类，是解释这些流为什么会撞到某种资源边界；第二类，是解释工程系统如何重写这些流，让边界不再以原来的方式出现。

所谓撞墙，最常见的无非是几种情况。内存流会撞上 HBM 容量和碎片化边界，于是系统要考虑分片、重计算、分页和外置。通信流会撞上机内与机间带宽、拓扑和同步开销，于是系统要考虑 collective、并行切分、路由和跨资源池传输。控制流会撞上调度粒度、依赖顺序和 host/runtime 开销，于是系统要考虑编译、融合、异步化与队列策略。数据流则会撞上输入管线、负载波动、长短不齐和版本切换，于是系统要考虑分桶、batch 组织、在线 admission 和流水线化。许多看起来不同的问题，归根到底只是某一类流先撞上了墙。

而所谓重写，就是系统设计真正开始发挥作用的地方。FSDP2、张量并行、流水并行、context parallelism 在改写的是训练中的内存流和通信流；`torch.compile`、kernel 融合、运行时捕获在压缩控制流中本可避免的调度空洞；FlexAttention 这类机制是在不牺牲注意力表达灵活性的前提下，重新安排计算与内存访问的关系；prefix reuse、continuous batching、paged KV cache 与 disaggregated serving 则是在改写 request 的内存流、通信流和控制流，让请求不必按最笨重的方式通过系统。到了 RL 后训练，问题会更明显地暴露为分布式数据流问题：rollout 何时产生、何时过滤、何时进入 trainer、何时做评估、何时保存状态，整个闭环像一条持续流动的生产线，而不是一次孤立优化。

> [!tip] 读法：后面每章都可以先问三个问题
> 第一，现在主导系统行为的是哪几种流。
> 第二，这些流先撞上了什么资源边界，表现成什么症状。
> 第三，当前这项技术到底在重写哪一种流，它靠什么把等待、占用或搬运改成更可接受的形状。

这也是为什么本书不能按框架名来组织。框架会变，API 会变，最佳实践也会变，但系统瓶颈不会凭空消失。训练侧始终在处理显存、通信和执行重叠；serving 侧始终在处理 KV cache、排队和调度；RL 后训练始终在处理 rollout 数据流、资源隔离与闭环稳定性。用“流如何撞墙、如何被重写”这个方法去读，就能把一串不断变化的框架名字还原成相对稳定的系统问题。

因此，系统视角的价值，不是给模型加一层工程包装，而是把训练、生成和后训练重新理解为受资源约束的动态过程。它要求我们不再只盯着某个局部算子、某个局部指标，而是去判断一段真实执行中哪些状态在流动、哪里形成了资源边界、哪些等待是不可避免的、哪些等待其实可以被工程地重写。

如果这一章的视角转换真正发生了，那么从下一章开始，读者看到的就不再是一堆分散技术，而是一条清晰主线：训练并行是在改写 step 的流，serving 系统是在改写 request 的流，RL 基础设施是在改写闭环的数据流，而所谓“大模型系统”，无非就是想办法让这些流在受限世界里持续地产生更多有用工作。
