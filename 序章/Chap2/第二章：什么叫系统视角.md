> 系统视角先做一件事：把“谁在算”换成“谁在流动，谁在等待”。

第一章把对象列出来了：参数、激活、优化器状态、KV cache、网络流量、调度与时间。第二章要做的，是把这些对象重新放回时间里。只知道系统里有什么，还不够。系统真正难的地方，在于这些对象会先后出现，会彼此挤压，会互相等待，也会在合适的时候被回收。

所以这一章先不往 GPU 细节里钻，也不急着去列更多术语。它先教一种看系统的方法。以后无论你面对训练并行、KV cache、continuous batching、checkpoint，还是 RL rollout pipeline，都可以先问三个问题：现在是什么在流动；它卡在了哪里；谁在等谁。能把这三个问题问清楚，后面的技术细节就会自己落位。

## 系统视角先换分析单位

从模型视角看，我们最熟悉的分析单位是层、算子、loss、梯度和参数更新。这个视角非常重要，因为它告诉我们模型在算什么、为什么会学到东西。可一旦进入大规模训练、在线 serving 和 RL 后训练，只看这些单位就不够了。系统里的很多问题，真正由状态能不能放下、能不能及时移动、能不能和别的步骤重叠来决定。

系统视角真正换掉的，是分析单位。它不再把“某一层的局部计算”当成默认起点，而是把“一次完整执行”当成起点。训练里，这个执行单位是一整个 step；serving 里，是一条 request 从进入到退出的生命周期；RL 里，则是 rollout、筛选、打分、update 组成的闭环。对象还是第一章那批对象，可一旦放回执行过程，它们就带上了时间结构：谁先出现，谁后出现，谁要常驻，谁只短暂停留，谁必须等别人准备好才能继续。

> [!abstract] 定义：系统视角里的分析单位
> 训练里，最小分析单位是一整个 step。
> serving 里，最小分析单位是一条 request 的完整生命周期。
> RL 里，最小分析单位是 rollout 到 update 的闭环。

拿参数举例。第一章里，我们先把它当成系统对象来认识；到了第二章，就要继续追问：它在 step 里何时被取用，何时被分片，何时被同步，何时被重组。KV cache 也是一样。第一章里，它只是 serving 的关键状态；到了这里，就得看它在 request 生命周期里怎样被建立、怎样延续、怎样命中已有前缀、怎样挡住或放行后面的请求。对象没有变，问题变了；原因就在于观察单位从静态对象换成了动态过程。

系统视角给模型研究补上的，正是这份“过程感”。模型视角擅长回答“这个结构为何有效”，系统视角擅长回答“这个执行为何变慢、为何放不下、为何扩不起来”。两种视角并不打架，它们只是看同一件事的两个面。第一章让我们知道系统里有什么，第二章要让我们看见这些东西怎样发生。

## 一次训练 step：把执行看成四种流

模型论文常把训练压缩成三步：forward、backward、update。这个写法没有问题，但它把系统里的很多代价藏了起来。对 AI Infra 来说，一次训练 step 更像四种流在同一段时间里同时展开：数据流、内存流、通信流、控制流。

> [!abstract] 定义：训练 step 的四种流
> 数据流：样本怎样被读取、打包、切分，再送入计算。
> 内存流：参数、激活、梯度、优化器状态何时出现，何时回收。
> 通信流：哪些对象必须跨设备交换、聚齐、同步或重分布。
> 控制流：执行顺序怎样安排，哪里能重叠，哪里必须等待。

先看数据流。样本不会自动变成适合计算的 batch。它们要先被读取、切分、过滤、对齐，再组织成 batch 或 microbatch，然后送到不同 rank 上。这个环节很容易被当成“训练前的准备工作”，其实它已经在 step 里面了。样本长短怎么分桶，microbatch 怎样切，输入怎样分发，都会改变后面的节奏。很多看上去像算力不足的问题，最后追下来，是数据没及时喂到位，或者数据组织方式让后续通信更难隐藏。

再看内存流。参数、激活、梯度、优化器状态、临时 buffer 都不是静止的“占用”。它们会在不同阶段出现，停留时间也不同。参数在前向和反向都会被访问，激活一般在前向产生、在反向消费，梯度在反向中逐步形成，优化器状态则在更新时被读写。于是训练里的显存问题首先是一个时间问题：哪些对象会在同一个时刻堆到一起，形成你承受不了的峰值。activation checkpointing 的作用，就在于改写这个时间安排，用更多计算换一条更平的显存曲线。

通信流只要进入多卡，就一定会出现。某些梯度要同步，某些参数要按需聚齐，某些激活或序列片段要在并行维度之间交换，checkpoint save / load 时又会有另一种状态搬运。要紧的地方不在于记住所有 collective 的定义，而在于先承认一件事：多卡训练从来不是把单卡训练复制几遍那么简单。它是在原有计算之外，再长出一条独立的消息路径。只要这条路径看不见，很多 step time 问题就解释不通。

最后是控制流。训练 step 常常会卡在执行关系里的等待上，不一定是某个巨慢的 kernel。哪些 microbatch 能先上，哪些通信能和计算重叠，哪些地方会形成全局同步点，哪些阶段会冒出 bubble，这些都属于控制流。所谓 bubble，本质上就是资源在那一小段时间里没有产出理想的有效工作。系统成熟不成熟，常常就看它能把这些空档压到多小。

把四种流放到同一张时间线上，训练就从“参数更新”变成了“过程协调”。后面章节里的很多技术，表面上彼此分散，实质上都在改写这四种流的形状：分片在改写内存流和通信流，编译与融合在压缩控制流里的小碎片，checkpoint 策略则在更长时间尺度上处理内存流和 I/O 的冲突。

## 一次生成 request：为什么 serving 不能只看成一段前向

如果训练系统容易被误会成“规模更大”，serving 系统最容易被误会成“删掉 backward 剩下来的那半段”。从计算图上看，这种理解好像有点道理；从系统过程上看，它会漏掉最关键的部分。因为一条 request 进入在线系统后，面对的是一段带状态、带排队、带调度的完整生命周期。

这条生命周期通常从队列开始。请求先被接纳，再和其他请求一起组成某种批次结构；随后进入 prefill，把整段 prompt 处理一遍，建立初始上下文；之后进入 decode，在已有历史上一个 token 一个 token 地继续生成；直到输出结束，相关状态被回收，这条 request 才算真正离开系统。

> [!abstract] 定义：prefill 与 decode
> prefill：对已有 prompt 做整段前向，建立初始上下文状态。
> decode：在已有历史上逐 token 继续生成，并持续复用已有状态。
> 两段工作都属于前向计算，但资源形态和调度目标并不一样。

把这个过程摊开来看，四种流会再次出现，只是轻重缓急和训练不同了。

数据流在 serving 里，指的是 request 本身怎样在系统里推进。它会排队，会进入 batch，会部分完成，也会继续留在系统里等待下一轮 decode。训练里的 batch 往往在 step 开始前就定好了，serving 里的 batch 却是活的。请求会不断进入、不断退出，这也是 continuous batching 之所以重要的原因。

内存流的中心也换了。训练时最显眼的是参数、激活、梯度和优化器状态；serving 时最显眼的往往是 KV cache。参数在推理阶段相对稳定，KV cache 的占用却会随着请求长度、上下文长度、并发和 prefix 命中率不断变化。于是 serving 的“内存问题”常常表现成缓存怎样分配、怎样复用、怎样迁移、怎样回收，而不是训练式的显存账本。

通信流在 serving 里，也很容易带上动态性。某个前缀已经缓存在哪台机器，请求该被路由到哪里，prefill 产生的 KV 要不要转给 decode 资源池，这些都会改写消息路径。尤其当服务做成 disaggregated 架构之后，通信不再只是“算完之后再发一点数据”，它本身已经成了 request 生命周期的一部分。

控制流在 serving 里比训练更显眼。系统需要不断决定谁先进入、和谁一起、等待多久、何时释放状态、何时让更短或更紧急的请求穿过去。batch window、队列策略、prefill / decode 的协作方式、prefix reuse 命中与否，都会直接改写 TTFT、ITL 和整体并发能力。也正因如此，serving 很像一个带状态的调度系统。

把这些问题放在一起看，生成就不该再被想成“训练里拿掉 backward 之后留下来的部分”。它有自己独立的内存问题、通信问题和调度问题。后面关于 KV cache、prefix reuse、continuous batching、disaggregated serving 的章节，研究的也正是这些问题怎样被工程地改写。

## 从这一章开始，后面所有章节都在研究“流如何撞墙、如何被改写”

只要把训练 step 和生成 request 都放进四种流的框架里，后面这本书的结构就会一下子清楚很多。后面的章节看起来在讲不同主题，实际做的事情大体只有两类：第一类，解释某种流为什么会撞到资源边界；第二类，解释工程系统怎样改写这条流，让边界不再以原来的方式出现。

所谓撞墙，最常见的就几类。内存流会撞上 HBM 容量和碎片化；通信流会撞上带宽、拓扑和同步代价；控制流会撞上依赖顺序、runtime 开销和调度粒度；数据流会撞上输入管线、负载波动和队列长度。许多表面上不一样的问题，往下追，常常只是某一类流先撞上了墙。

所谓改写，就是系统设计真正开始起作用的地方。FSDP2、张量并行、流水并行、context parallelism 在改写训练里的内存流和通信流；`torch.compile`、kernel 融合、运行时捕获在压缩控制流里的碎片和空档；prefix reuse、continuous batching、paged KV cache、disaggregated serving 在改写 request 的内存流、通信流和控制流；到了 RL 后训练，rollout、过滤、trainer、评估和恢复接成一条闭环，问题就更像显式的数据流系统了。

> [!tip] 读法：后面每章都可以先问三个问题
> 第一，当前主导系统行为的是哪几种流。
> 第二，这些流先撞上了什么资源边界，表现成什么症状。
> 第三，这项技术到底在改写哪一种流，它怎样把等待、占用或搬运变成更能接受的形状。

所以系统视角的价值，不在于给模型再包一层工程术语，而在于把训练、生成和后训练重新放回动态过程里。它要求我们看见真实执行，看见等待，看见资源边界。这样一来，后面的训练并行、serving、KV cache 和 RL 数据流就不再是散点，而会串成一条线。
