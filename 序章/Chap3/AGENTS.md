第 3 章最好写成一章**“读这本书的操作系统”**：它不讲某类具体对象，也不讲某类具体硬件，而是给读者一套贯穿全书的研究顺序和判断标准。这样它和第 1 章“研究对象”、第 2 章“系统视角”是连续的：第 1 章回答“研究什么”，第 2 章回答“怎么看执行过程”，第 3 章回答“遇到一个 AI Infra 问题时，应该按什么顺序思考和动手”。这也和你前面确定的全书定位一致：按系统瓶颈来组织，而不是按框架名堆知识。

下面给一个更正式、适合直接放进目录的提纲。

# 第 3 章：本书的方法论

## 本章定位

前两章已经完成了两件事。第一章定义了 AI Infra 的研究对象：参数、激活、优化器状态、KV cache、网络流量、调度与时间。第二章则进一步说明，这些对象一旦进入一次训练 step 或一次生成 request，就会形成数据流、内存流、通信流和控制流。到了第三章，问题不再是“对象是什么”或者“过程怎么展开”，而是：**当我们真的面对一个系统问题时，应该先看什么，后看什么，如何避免一开始就被框架名、局部技巧和理论口号带偏。**

因此，本章不是知识章节，而是方法章节。它要建立的不是新的系统对象，而是一套研究顺序。全章可以用三句话概括：

先建立瓶颈模型，再学框架。  
先做 profiling，再谈优化。  
先看 wall-clock，再看理论复杂度。

---

## 3.1 为什么 AI Infra 需要方法论

这一节的作用，是先把“为什么这一章存在”讲明白。

AI Infra 很容易让初学者陷入三种常见偏差。第一种偏差，是把框架当成知识结构本身，学到最后记住了许多 API 和系统名，却没有形成对瓶颈的统一理解。第二种偏差，是看到一个优化技巧就立刻套用，却没有先确认系统是否真的卡在那个地方。第三种偏差，是被理论复杂度、paper benchmark 或局部 kernel 指标吸引，却没有先问整体 wall-clock 是否真的变快。

因此，这一章的任务，是建立本书默认的研究次序：  
先判断系统主要受什么约束；  
再用 profiling 把这个判断落到证据上；  
然后才进入优化与框架层面的实现选择。

这一节最后可以收束成一句定义：

**本书的方法论，不是从工具出发理解系统，而是从瓶颈出发选择工具。**

---

## 3.2 第一原则：先建立瓶颈模型，再学框架

这是本章第一块主体，也是最重要的一条原则。

这一节一开始要明确：框架只是某类系统假设的实现形式，而不是理解系统的起点。FSDP、Megatron、vLLM、Dynamo、verl、OpenRLHF 这些名字都很重要，但如果读者没有先建立“我现在面对的到底是显存瓶颈、通信瓶颈、调度瓶颈还是 I/O 瓶颈”的判断，框架知识就会变成散乱的产品说明书，而不是研究能力。这个取向和你第一章里强调的“AI Infra 研究的是状态如何被放置、移动、调度与恢复”是一致的。

这节建议分成三个连续问题来写。

第一个问题是：什么叫瓶颈模型。  
这里不要把它写成形式化数学模型，而要把它定义成一种系统压缩描述：在某一阶段、某一配置下，决定整体表现的主导约束是什么。一个好的瓶颈模型，不要求一开始就精确到 kernel 级别，但必须能回答：系统主要卡在算力、显存容量、内存带宽、设备间通信、host 调度、请求队列，还是 checkpoint / 数据输入这样的外围路径。

第二个问题是：为什么瓶颈模型要先于框架。  
因为同一个框架可以解决的问题边界是有限的，而真实系统往往是多重约束叠加。如果本质是 KV cache 占用和在线调度问题，那么一味研究训练框架没有意义；如果本质是 checkpoint 和恢复路径拖慢训练，那么单纯追求更高算子吞吐也抓不住主问题。框架只是把某种瓶颈上的经验做成可复用结构，而不是替代你做判断。

第三个问题是：如何粗粒度地建立瓶颈模型。  
这里可以给出本书默认的最小问题表述方式：  
训练里先问，当前主要受限于内存、通信、计算、I/O 还是控制开销；  
推理里先问，当前主要受限于 TTFT、ITL、KV cache、batching、prefill/decode 失衡还是队列；  
RL 里先问，当前主要受限于 rollout、update、数据等待、资源空转还是系统恢复。  
这一节不需要展开具体指标推导，只要把“先判断主导约束”这个思维植入进去。

这一节最后最合适的结论是：

**系统学习不应该从“有哪些框架”开始，而应该从“我需要解释哪种瓶颈”开始。**

---

## 3.3 第二原则：先做 profiling，再谈优化

这是第二块主体，建议写得非常坚决。

这节要先反复强调一个事实：没有 profiling 的优化，大多数时候只是把个人偏好施加到系统上。系统优化不是“把自己熟悉的技巧轮流试一遍”，而是先让系统暴露证据，再沿着证据动手。这里刚好和后面第一部“硬件、运行时与 profiling”形成很强的桥接关系：那一部会教读者怎么读 timeline、怎么看 kernel metric、怎么找热点；而第三章先把这件事上升为方法论纪律。

这节内部可以围绕三个层次展开。

第一个层次，是为什么 profiling 比直觉更重要。  
因为 AI Infra 的很多现象是反直觉的。GPU utilization 高不一定说明系统健康；通信多不一定真是瓶颈；某个 kernel 耗时大也不一定就是整体 wall-clock 的主因。训练系统可能卡在同步点，推理系统可能卡在排队和状态迁移，RL 系统可能卡在资源池错配。仅凭经验判断，常常会优化错对象。

第二个层次，是 profiling 到底是在找什么。  
不是单纯找“最慢的 kernel”，而是找证据链。一个好的 profiling 过程，应该能回答：时间到底花在哪，等待发生在哪，重叠是否成立，真正决定 wall-clock 的临界路径是什么。换句话说，profiling 的目标不是收集更多图表，而是识别系统的关键路径。

第三个层次，是 profiling 和优化之间的顺序关系。  
优化只有在 profiling 之后才有意义，因为你必须先知道自己是在缩短关键路径，还是只是在改善一个本来就不重要的局部。一个算子提升 20%，如果它不在关键路径上，最终 wall-clock 可能几乎不变；一个小小的调度调整，如果能减少长尾等待，整体吞吐和延迟反而会显著变化。

这一节最后可以收住为：

**优化不是从技巧开始，而是从证据开始。profiling 不是辅助工作，它本身就是系统研究的一部分。**

## 3.4 第三原则：先看 wall-clock，再看理论复杂度

这是本章最有“立场”的一节，也很适合作为这本书的方法论招牌。

这一节开头就可以直接说：理论复杂度当然重要，但在系统研究里，它不是第一观察量。一个系统首先要回答的，不是“渐进上更优吗”，而是“真实机器上更快吗，稳定吗，成本更低吗”。这是因为复杂度只描述运算量的某种抽象增长关系，而 wall-clock 才把计算、内存、通信、调度、缓存命中、重叠失败、恢复开销这些真实因素统一到一个结果变量里。

这节建议分成三个部分写。

第一部分，为什么 wall-clock 是系统中的第一事实。  
因为用户感受到的是训练一天出了多少 step、服务首 token 等多久、后训练一轮 rollout + update 要多久，而不是某个模块在 paper 里从 O(n²) 变成了 O(n log n) 或 O(n)。系统世界里，最终要落到时间、吞吐、延迟和成本上。

第二部分，为什么理论复杂度仍然重要，但位置更靠后。  
复杂度不是没用，而是它应该在你已经确认 wall-clock 问题之后，用来解释趋势、外推规模、判断某种方法在更大上下文或更大并行度下是否有潜力。它更像“第二性解释变量”，而不是第一性证据。

第三部分，AI Infra 中常见的复杂度错觉。  
有些方法在理论上减少了 FLOPs，却引入更多内存访问和控制开销，结果 wall-clock 反而更差；有些方法理论上更优，但只在某个序列长度或某种硬件代际下才开始体现收益；还有些方法 paper 上复杂度更漂亮，但工程实现导致 graph break、shape specialization 失败、kernel 退化，结果真实收益被吃掉。这一节不需要大量实例，只要把读者的判断习惯扭过来即可。

这一节最后可以形成一句全章最有辨识度的话：

**在 AI Infra 里，理论复杂度是解释，wall-clock 才是判决。**

---

## 3.5 把三条原则合在一起：本书默认的研究顺序

前面三节分别立了三条原则，这一节要把它们合成一个真正可执行的工作流。这样第三章就不只是态度表述，而是能指导后面每一章的阅读与实践。

这里建议把本书的默认研究顺序写成一个连续方法，而不是分散提示：

第一步，先界定场景。是训练、生成还是 RL 后训练；是单机还是分布式；是吞吐问题、延迟问题还是恢复问题。  
第二步，建立粗粒度瓶颈模型。先判断主导约束落在哪一类资源或哪一类流上。  
第三步，做 profiling。确认关键路径、等待位置和真正影响 wall-clock 的因素。  
第四步，再选择优化方向。决定应该改内存流、通信流、控制流还是数据流。  
第五步，最后才进入框架和实现。此时框架是为瓶颈服务的，而不是反过来支配问题定义。

这一节里可以明确告诉读者：后面这本书的所有章节，默认都会按这个顺序组织。先解释问题在哪，再介绍人们如何解决它，而不是一上来先讲“某某框架有哪几个 feature”。

这一节最好用一个短结论收住：

**方法论的价值，在于它让不同章节共享同一种思考顺序。**

## 3.6 本章与后续章节的关系

这一节作为桥接，写短一点即可。

后面的第一部“硬件、运行时与 profiling”，是在给第二原则提供工具基础：教你怎样看到证据。  
后面的训练并行与编译章节，是在给第一原则提供实现空间：教你如何重写瓶颈所在的流。  
后面的 serving 与 KV cache 章节，是在把第三原则落到在线系统：教你如何用 wall-clock 和 SLA 约束去判断优化是否真的成立。  
后面的 RL Infra 章节，则会把这一整套方法搬到更长链条的数据流系统上。

这样，第三章就会成为全书真正的“方法论中枢”。

## 本章总结

这一章不增加新的系统对象，而是规定一套研究顺序。它要反复纠正的，是三种在 AI Infra 里极其常见的误区：先记框架，再找问题；先试优化，再看证据；先谈复杂度，再看真实时间。

本书的方法论恰好相反：  
先建立瓶颈模型，再学框架。  
先做 profiling，再谈优化。  
先看 wall-clock，再看理论复杂度。

一旦这三条顺序被建立起来，后面的 GPU、内存层级、kernel、collective、并行、KV cache、serving 和 RL rollout，就都不再是零散技巧，而会成为同一种系统研究方法在不同对象上的展开。
