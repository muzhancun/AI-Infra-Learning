> 真正有用的方法论，不是替你提供更多结论，而是规定结论必须经过怎样的顺序才有资格成立。

第一章回答的是“研究什么”：AI Infra 研究的不是层本身，而是参数、激活、优化器状态、KV cache、网络流量、调度与时间这些系统对象。第二章回答的是“怎么看执行过程”：这些对象一旦进入一次训练 step 或一次生成 request，就会沿着数据流、内存流、通信流和控制流在时间中展开、冲突、等待与回收。可即便前两步都已经建立起来，读者真正面对一个系统问题时，仍然很容易在一开始就走错方向。因为真实世界里的 AI Infra 从来不缺框架名、paper 结论、benchmark 图表和局部优化技巧，缺的往往是一个稳定的起手顺序。

第三章要补上的，正是这件事。它不再增加新的系统对象，也不再扩展新的执行流，而是给整本书设定一套默认的研究纪律：遇到一个问题时，应该先看什么，后看什么，哪些判断必须晚一点做，哪些判断必须先立住。换句话说，这一章不是知识章节，而是操作系统章节。它要告诉读者的不是“世界上有哪些方案”，而是“当方案很多、现象很乱、局部信号彼此矛盾时，应该按什么顺序组织自己的观察与动作”。

本章的全部内容，其实可以收束成三条顺序原则。第一，先建立瓶颈模型，再学框架。第二，先做 profiling，再谈优化。第三，先看 wall-clock，再看理论复杂度。它们听上去像三句口号，但真正重要的地方不是句子本身，而是其中的先后关系。AI Infra 的多数误判，都不是因为人完全不知道这些概念，而是因为顺序错了：先记框架，再找问题；先试优化，再看证据；先谈复杂度，再看真实时间。第三章的任务，就是把这个顺序扭回来。

## 为什么 AI Infra 需要方法论

AI Infra 比许多技术领域更容易制造“理解已经发生了”的错觉。框架名会制造这种错觉，仿佛只要记住 FSDP、Megatron、vLLM、Dynamo、verl 这些名字，就已经掌握了系统世界的版图；局部优化会制造这种错觉，仿佛只要把自己熟悉的技巧轮流试一遍，总能逼近正确答案；理论复杂度也会制造这种错觉，仿佛只要 paper 上的增长阶更漂亮，真实系统就会自动跟着受益。可系统问题恰恰最擅长揭穿这种表面上的熟悉感。一个训练系统可能真正卡在 checkpoint save/load 与恢复链路上，一个 serving 系统可能真正卡在队列和 KV 驻留上，一个 RL 后训练系统可能真正卡在 rollout 与 update 的错配上。只盯着最显眼的框架、最顺手的技巧、最漂亮的理论式子，最后常常只是把症状当成原因。

因此，方法论在 AI Infra 里并不是“写作上的美化词”，而是抵抗错觉的纪律。它的作用，不是让你瞬间更聪明，而是防止你在错误的地方开始聪明。你必须先有一个对主导约束的粗判断，才能知道应该读哪类系统、看哪类指标、怀疑哪类路径；你必须先让系统暴露证据，才能知道自己的优化究竟是在缩短关键路径，还是只是在抚平一个不重要的局部波动；你必须先回到真实 wall-clock，才能判断一个理论上更优的方法在现实机器上到底有没有成立。方法论真正提供的，不是答案，而是答案出现之前的秩序。

> [!abstract] 本章要反复纠正的三种顺序错误
> 第一，先记框架，再找问题。  
> 第二，先试优化，再看证据。  
> 第三，先谈复杂度，再看真实时间。  
> 本章的方法论，就是把这三个顺序倒过来。

如果把第一章和第二章比作“对象表”和“过程图”，那么第三章就是“阅读与行动说明书”。从这一章开始，后面每一章都不只是告诉你一个系统技术“是什么”，而是在默认你已经接受一套统一顺序：先问主导瓶颈，再找证据链，再判断 wall-clock 是否真的改善，然后才进入框架与实现细节。只要这个顺序建立起来，本书后面的并行、编译、KV cache、serving 与 RL 基础设施，就会共享同一种阅读方式。

## 先建立瓶颈模型，再学框架

系统研究最容易犯的第一个错误，就是把框架当成问题定义本身。可框架从来不是理解系统的起点，它只是某种系统假设被沉淀出来的实现形式。一个人之所以会觉得框架知识越来越碎，往往不是因为框架太多，而是因为他在读框架之前，还没有先回答：我现在面对的究竟是哪一种主导约束。是显存放不下，还是带宽跟不上；是通信同步压住了 step time，还是 host 调度让 pipeline 出现气泡；是 KV cache 驻留与回收出了问题，还是请求排队让 TTFT 被拖长；是 rollout 供给不足，还是 trainer 在等数据，抑或两边资源池彼此空转。只要这些问题还没立住，框架就会从解决方案退化成名词表。

> [!abstract] 定义：瓶颈模型
> 瓶颈模型不是形式化推导得很完整的数学模型，而是对某一阶段、某一配置下主导约束的压缩描述。
> 一个合格的瓶颈模型，至少要能回答：系统主要受限于计算、容量、带宽、通信、I/O、调度，还是队列与状态驻留。

瓶颈模型之所以必须先于框架，是因为框架解决问题的边界从来都是局部的。FSDP2 之类的训练框架，把参数、梯度和优化器状态视为可以分片、重组和按需聚齐的对象，它在本质上是在回应训练中的状态放置与通信平衡问题；Megatron Core 的并行技术，则是在更细粒度地重写计算、状态与通信如何沿不同维度展开；vLLM 和 Dynamo 关心的是在线 serving 里 KV cache、batching、prefill/decode 拆分、路由与缓存驻留；而 RL 后训练框架越来越像数据流系统，试图处理的是 rollout、过滤、训练、评估和恢复如何稳定地接成闭环。每一个框架背后都隐含了一组系统假设。可如果读者没有先判断自己到底在解释什么，框架知识就会变成散乱的产品说明书，而不是研究能力。

这也是为什么瓶颈模型不必一开始就很精确，却必须一开始就存在。系统分析的第一步，往往不是立即深入到 kernel 或 runtime，而是先在粗粒度上把主要矛盾分出来。训练里，先问当前主导问题是在内存、通信、计算、I/O 还是控制开销；推理里，先问问题主要落在 TTFT、ITL、KV cache、prefill/decode 失衡、batching 还是队列；RL 里，先问系统主要受限于 rollout、update、数据等待、资源空转还是恢复路径。这个阶段允许你暂时模糊，但不允许你完全没有判断。因为没有瓶颈模型，后面的 profiling 也会失去方向：你会收集到很多图，却不知道哪一些图真正值得看。

真正成熟的系统学习，总是从“我要解释哪一种瓶颈”开始，而不是从“当前流行哪一个框架”开始。框架当然重要，但它应该在问题已经被初步定义之后进入。一个系统研究者先要学会区分问题形态，再去学习哪种实现曾经在某类问题形态上有效。只有这样，框架才会从外部世界的噪声，变成你手里的结构化工具。

## 先做 profiling，再谈优化

如果说瓶颈模型解决的是“问题该朝哪里看”，那么 profiling 解决的就是“这个判断如何从直觉变成证据”。AI Infra 的第二个常见错误，是一旦怀疑某个地方可能是瓶颈，就马上开始优化。可没有 profiling 的优化，大多数时候只是在把个人偏好强加给系统。熟悉 kernel 的人会先想算子融合，熟悉分布式的人会先想通信重排，熟悉编译的人会先想图捕获，熟悉服务的人会先想调队列。每一种直觉都可能对，也都可能错。真正危险的，不是某种直觉本身，而是它在没有证据时就开始主导行动。

profiling 的价值，不在于让你得到更多图表，而在于让系统形成证据链。你需要知道时间到底花在哪里，等待究竟发生在哪里，哪些计算与通信真的重叠了，哪些重叠只是想象中的重叠，真正决定 wall-clock 的临界路径是什么。只有这些问题被回答，优化才有资格开始。否则，一个局部算子提速 20%，如果它根本不在关键路径上，最终 step time 或 E2E latency 可能几乎不动；反过来，一个看上去不起眼的队列调整、同步点消除或缓存命中提升，如果它恰好切中了关键路径，带来的 wall-clock 改善反而会远大于某个更“硬核”的局部微优化。

> [!abstract] 定义：关键路径
> 关键路径不是“最慢的那个 kernel”，而是决定一次 step、一次 request 或一段闭环总时长的那条临界执行链。
> profiling 的目标，不是找最大的局部耗时，而是确认系统的关键路径究竟由哪些等待、计算、通信和调度关系组成。

这也是为什么 profiling 在本书里不是一项辅助技能，而是一种方法论纪律。系统世界里有太多反直觉现象：GPU utilization 高，并不等于系统健康，因为它可能忙的是同步等待或低价值工作；通信量大，也不自动意味着通信就是瓶颈，因为有些通信可能已经被有效隐藏；某个 kernel 的耗时占比很高，也不必然说明它值得最先优化，因为整体 wall-clock 也许更多被排队、图断裂、host 调度或状态迁移决定。profiling 的意义，恰恰是把这些表面信号重新放回时间线上，让“谁在等谁”从印象变成事实。

因此，优化和 profiling 之间不是并列关系，而是前后关系。优化从来不该先于 profiling，因为你必须先知道自己是在缩短关键路径，还是只是在修饰一段并不主导总时长的局部。后面第一部“硬件、运行时与 profiling”之所以先出现，也正是因为它要为这条方法论原则提供工具基础：先教你怎样看见证据，再谈如何根据证据改写系统。

## 先看 wall-clock，再看理论复杂度

系统研究最有诱惑力、也最容易让人失焦的第三种对象，是理论复杂度。复杂度当然重要，它能帮助人理解趋势、判断扩展性、预估规模变化后的增长关系；可在 AI Infra 里，它不是第一观察量。一个系统首先必须回答的，不是“渐进上更优吗”，而是“真实机器上更快吗，稳定吗，成本更低吗”。因为复杂度只描述了运算量或资源需求的某种抽象增长关系，而 wall-clock 才把计算、内存访问、设备间通信、调度开销、缓存命中、重叠失败、恢复路径这些现实因素统一进同一个结果变量里。

> [!abstract] 定义：wall-clock
> wall-clock 指从外部视角观察，一个真实任务从开始到结束实际经过了多久。
> 在系统里，它是把计算、通信、内存、调度、I/O 与恢复成本统一到一起的第一事实。

这也是为什么系统里的“第一判决”必须交给 wall-clock。用户真正感受到的，是一天能跑多少有效 step，是请求首 token 要等多久、后续 token 是否平稳，是一轮 rollout 加 update 要消耗多少真实时间，而不是某个模块在 paper 上从 \(O(n^2)\) 变成 \(O(n \log n)\) 或 \(O(n)\)。理论复杂度可以解释趋势，却不能代替真实执行。一个方法理论上减少了 FLOPs，却引入了更多内存访问和控制开销，wall-clock 可能反而更差；一个方法在大规模极限下更优，却只有当序列长度、batch 规模或硬件代际跨过某个阈值之后才显出优势；还有一些方法在理论上无可挑剔，但工程实现导致 graph break、shape specialization 失败、缓存局部性变差，最后真实收益被执行细节吃掉。复杂度并没有错，错的是把它过早地拿来代替判决。

这并不意味着复杂度不重要。恰恰相反，复杂度在系统里有一个非常稳定、也非常有价值的位置：它是第二性的解释变量。当你已经确认某个问题在 wall-clock 上确实存在，复杂度可以帮助你理解它为什么会在规模扩大时变得更严重，也可以帮助你判断某种解法在更大上下文、更高并行度或更长时间跨度下是否仍有潜力。换句话说，复杂度更像“解释趋势与外推边界的工具”，而不是“宣布方法已经有效的证据”。

所以在 AI Infra 里，正确的顺序不是抛弃复杂度，而是把复杂度放回它该在的位置：先看 wall-clock，再看复杂度；先确认现实是否改善，再讨论抽象上为什么可能改善。只要这个顺序不乱，理论和系统就不会互相拆台，而会变成彼此补充的两种语言。

## 把三条原则合在一起：本书默认的研究顺序

前面三节分别立了三条原则，但它们真正有力量的时候，不是分开记忆，而是被合成为一条可执行的工作流。因为系统问题从来不是三道独立选择题，而是一串连续动作：你先得知道自己在什么场景里，再初步判断主导约束落在哪里，再用 profiling 把这个判断压实成证据，之后才谈该改哪一类流、选哪一种实现。只要顺序乱掉，后面的动作就会越来越像碰运气。

因此，本书默认的研究顺序可以写得非常明确。第一步，先界定场景与目标：这是训练、生成还是 RL 后训练，是吞吐问题、延迟问题还是恢复问题，是单机局部现象还是分布式扩展现象。第二步，建立粗粒度瓶颈模型：先分清主导约束主要落在计算、内存、通信、I/O、调度、队列还是状态驻留。第三步，做 profiling：确认关键路径、等待位置、重叠是否成立，以及真正决定 wall-clock 的因素。第四步，再选择优化方向：判断该改写的是数据流、内存流、通信流还是控制流。第五步，最后才进入框架与实现：此时框架不再支配问题定义，而是被当成服务于瓶颈的具体工具。

> [!tip] 本书默认的读法
> 先问：我现在面对的是哪一种场景，目标指标是什么。  
> 再问：主导瓶颈大致落在哪一类资源或哪一种流上。  
> 然后用 profiling 把这个判断变成证据。  
> 最后才问：该选哪类优化与哪套框架实现。

这套顺序还有一个更深的价值：它让后面各章共享同一种思考方式。训练并行、编译、KV cache、serving 调度、RL rollout pipeline 表面上谈的是不同对象，实际上都可以被放回这一条顺序链里。你不会再一上来问“这个框架有哪些 feature”，而会先问“它试图解决的瓶颈是什么”“它重写了哪一种流”“它有没有真的改善 wall-clock”。这样一来，后面每一章都不再只是某项技术的介绍，而是同一种系统方法在不同对象上的展开。

## 从这一章开始，后面章节共享同一种思考顺序

第三章真正想建立的，不是一个漂亮口号，而是一种贯穿全书的阅读秩序。后面的第一部“硬件、运行时与 profiling”，是在给第二原则提供工具基础：它教你怎样看见证据，怎样区分算力问题、带宽问题、同步问题与调度问题。再后面的训练并行与编译章节，是在给第一原则提供实现空间：当瓶颈被确认后，系统如何通过分片、重组、融合、捕获与调度去重写那条出问题的流。serving 与 KV cache 章节，则把第三原则推到在线系统里：在那里，真正有资格裁决优化是否成立的，不是 paper 里的复杂度，而是 TTFT、ITL、E2E latency、并发与 SLA 约束下的真实 wall-clock 表现。到了 RL 基础设施章节，同样的方法会被搬到更长链条的数据流系统上，去判断 rollout、update、版本同步与恢复路径究竟谁在拖慢闭环。

也正因为如此，第三章虽然不引入新的系统对象，却可能是全书最关键的一章。前两章解决的是“看见什么”，这一章解决的是“按什么顺序看”。一旦顺序建立起来，后面的 GPU、HBM、kernel、collective、并行、KV cache、serving 与 RL 后训练，就都不再是分散技巧，而会成为一套统一系统方法在不同场景中的连续展开。

所以，本章最后真正要留下来的，不是若干散句，而是一个稳定习惯：遇到问题时，先建立瓶颈模型，再学框架；先做 profiling，再谈优化；先看 wall-clock，再看理论复杂度。只要这三条顺序被建立起来，AI Infra 就不再只是不断变动的框架名和技巧库，而会重新显出它作为系统科学的骨架。
