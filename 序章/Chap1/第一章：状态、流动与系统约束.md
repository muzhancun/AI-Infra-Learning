> 大模型系统可以先当成一件很朴素的事来看：许多状态被放在有限的显存和带宽里，按时间顺序被取出、移动、计算，再放回去。

当模型进入大规模训练、在线服务和 RL 后训练，问题就变了。数学形式常常没有大改，真正决定系统能不能跑、跑得稳不稳、贵不贵的，往往是另一组更具体的事情：参数放不放得下，激活顶不顶得住，KV cache 能不能复用，请求怎样排队，checkpoint 会不会拖住主路径，故障之后多久能恢复。把这些问题放在一起看，AI Infra 研究的其实是一件很直接的事：在显存、带宽、延迟和可靠性约束下，怎样让有用 token 持续流过系统。

到 2026，这条主线已经很清楚。训练侧常见的是 FSDP2、DCP、`torch.compile`、FlexAttention、Megatron Core；推理侧绕着 KV cache、prefix caching、continuous batching、disaggregated serving 打转；RL 后训练越来越像分布式数据流系统，重点落在 rollout、过滤、学习、评估和恢复能不能顺畅接起来。框架名字可以很多，但它们关心的还是同一批系统问题：状态放在哪，什么时候移动，谁来调度，代价落在何处。

> [!abstract] 定义：本书里的 AI Infra
> 本书所说的 AI Infra，专指训练、推理和后训练中状态对象如何被放置、搬运、调度、缓存和恢复。
> 它最关心四类约束：显存、带宽、延迟、可靠性。
> 一旦把这些约束放到前台，系统研究就有了自己的对象和语言。

## 从模型对象到系统对象

只从模型视角出发，我们最熟悉的是层、算子、张量形状和 loss。这些东西当然重要，但还不够。系统视角关心的是另一组对象：参数、激活、优化器状态、KV cache、网络流量、调度和时间。

- 参数决定模型能否放下，也决定分片和同步要付出多少代价。  
- 激活决定长上下文和大 batch 能不能成立。  
- 优化器状态决定训练显存和 checkpoint 体积。  
- KV cache 决定推理时的内存经济学。  
- 网络流量决定多卡多机能扩多远。  
- 调度决定资源有没有空转。  
- 时间把前面所有对象放进 step、request、episode 这三条轴线里。

这里的 step 是一次训练更新，request 是一条在线推理请求，episode 是 RL 里从采样到回报的一段完整轨迹。三条时间轴不一样，状态的生命周期也就不一样。参数在训练里要反复取用，KV cache 会跟着请求存活一段时间，rollout 产生的数据还要穿过筛选、打分和 update 才算真正用上。系统工作的难点，就在这些对象既拥挤，又彼此依赖。

> [!abstract] 定义：系统对象
> 在本书里，系统对象指那些必须被保存、搬运、同步、缓存或恢复的状态。
> 最常见的一组对象是：参数、激活、优化器状态、KV cache、网络流量、调度和时间。
> 许多系统瓶颈，本质上都是这些对象的生命周期撞在了一起。

## 训练时真正占内存和时间的东西

把一次训练 step 写成账本，很多问题就清楚了。这个账本至少有六类东西：参数、梯度、优化器状态、激活、临时 buffer / 通信 staging、checkpoint I/O。

这里的临时 buffer，可以把它想成算子运行时借来的工作台；GEMM、attention 或 fused kernel 往往都要临时占一块地方。通信 staging 则像打包区：all-gather、reduce-scatter 这类操作前后，张量经常要先拼好、先放稳，再送上路。训练能不能扩起来，往往取决于这张账本在一个 step 里怎样展开：谁先出现，谁占得久，谁能回收，谁又必须和别人重叠。

> [!abstract] 定义：训练 step 的状态账本
> 一次训练 step 不只包含前向、反向和更新。
> 它还包含状态的驻留、临时 workspace、通信准备、checkpoint 保存与恢复。
> 所谓训练优化，常常就是在重写这张账本的时间安排。

FSDP2 的价值，用一句很直白的话说，就是把训练状态切开来放，只在需要的时候再拼回去。传统数据并行很像每张卡都背一整本字典；FSDP2 更像每张卡只背自己负责的页，轮到某一页时大家再临时把它拼齐。这样一来，参数、梯度和优化器状态不必长期完整地躺在每张卡上。Megatron Core 的 context parallelism 又提醒我们，激活也能沿序列维拆开来放；很长的上下文，不一定非要压在单卡的完整序列上。

`torch.compile` 和 FlexAttention 把另一层问题拉了进来：计算图本身也会影响训练效率。编译器能不能抓住一整段图，能不能把多个小操作并成更大的执行块，动态 shape 会不会把本来可以复用的编译结果打散，这些都会直接反映到 wall-clock 上。这里常见的一个词是 graph break。它可以简单理解成：本来希望被编译成一个整体的执行过程，在某处断开了，又退回到零散的 eager 执行。断得太多，编译收益就会很快缩水。

> [!abstract] 定义：FSDP2、DCP、graph break
> FSDP2：把参数、梯度和优化器状态按 rank 分片，计算时再按需聚齐。
> DCP：Distributed Checkpoint，多 rank 并行保存和加载训练状态，也支持重新切分。
> graph break：一段本来想整体编译的计算图中途断开，重新回到 eager 执行。

DCP 把另一件容易被忽略的事情摆到了台面上：checkpoint 不是训练结束时顺手存一下的附件，它就是训练路径的一部分。训练会中断，会失败，会迁移机器，也会换拓扑。只要这些事会发生，save、load、restart 的成本就会进入系统账本。异步保存的意义也很朴素：尽量把写盘和主路径拆开，不让恢复能力把训练吞吐压得太狠。

> [!tip] 直觉：为什么 checkpoint 要放进主路径里看
> 训练只要会中断，checkpoint 就一定会变成系统成本。
> 它决定了失败之后多久能恢复，恢复时要搬多少状态，也决定 save / load 会不会拖慢正常训练。

## 推理系统为什么围着 KV cache 转

训练时最显眼的是参数和激活，推理时最显眼的往往是 KV cache。原因不复杂：参数在推理阶段相对稳定，真正随着请求长度、并发和调度方式不断变化的，是历史 token 留下来的那份状态。

> [!abstract] 定义：KV cache
> KV cache 是自回归推理时为历史 token 保存的 key / value 表示。
> 有了它，模型生成下一个 token 时就不用把整段上下文重算一遍。
> 它省下了大量计算，也把内存管理推到了系统中心。

一条 request 进入系统以后，通常先排队，再进入 prefill，把整段 prompt 跑一遍，建立最初的上下文状态；然后进入 decode，在已有历史上一个 token 一个 token 地继续生成。只要把这个过程完整地看一遍，就会发现推理系统的核心工作很少是“把前向做快一点”这么简单。它更像在处理一件持续发生的调度问题：哪些请求该先进去，哪些请求能一起组成 batch，哪些前缀能直接复用，哪些 KV 该留在 GPU，哪些 KV 可以先挪走。

这也是为什么今天很多 serving 系统都把 continuous batching、prefix caching、PagedAttention、running / waiting requests、GPU cache usage、TTFT、ITL、prompt tokens/s、generation tokens/s 直接当成日常监控项。continuous batching 的意思是，请求不会整批同时开始和结束，它们会在运行中不断进入和退出同一个批次。prefix caching 的意思是，几条请求如果共享前缀，就直接复用那段已经算好的 KV。PagedAttention 可以粗略看成“把 KV cache 按页来管”，这样系统不用总去找大块连续显存，碎片也会少很多。

> [!abstract] 定义：continuous batching、prefix caching、PagedAttention
> continuous batching：请求在运行过程中动态进入和退出同一个 batch。
> prefix caching：共享前缀的请求直接复用已经算好的前缀 KV。
> PagedAttention：把 KV cache 按页组织，减轻碎片和大块连续分配压力。

一旦服务被拆成 prefill 和 decode 两段，很多原来模糊的系统问题都会变得可见。prefill 更像大吞吐的矩阵计算，decode 更像小步快跑的低延迟循环。两段工作的资源形态不同，调度目标也不同，所以越来越多的系统会把它们分开看，甚至分开部署。KV-aware routing、KV offloading、分层缓存这些设计，最后都落在同一个问题上：哪份状态该驻留在哪，什么时候动，什么时候回收，才能在给定的 SLA 下把更多请求稳稳接住。

> [!tip] 直觉：为什么 prefill 和 decode 常常要拆开看
> prefill 追求大吞吐，decode 追求小延迟。
> 两段工作放在同一种资源和同一种调度策略里，通常很难同时做到理想状态。

## AI Infra 关心的指标地图

系统研究离不开指标，但不同场景要看不同仪表盘。训练侧最先该看的通常是 MFU、step time、tokens/s、峰值显存、checkpoint save / load / restart 开销，以及编译命中情况和 graph break。MFU 可以理解成“理论上能做的模型计算里，真正变成有效工作的那部分占了多少”；step time 则更像总账，它把计算、通信、等待和 I/O 一起折算进来。

> [!abstract] 定义：训练侧最常看的三个指标
> MFU：模型有效 FLOPs 占理论可用 FLOPs 的比例。
> step time：一次参数更新从开始到结束的总时长。
> 峰值显存：某个时刻的最大 HBM 占用，决定配置能不能真的跑起来。

推理侧的仪表盘要贴着用户体验来建。TTFT 是 time to first token，也就是用户多久能看到第一个 token；ITL 是相邻两个输出 token 之间的间隔；TPOT 常用来表示平均每个输出 token 的时间；E2E latency 记录的是一条请求从进来到出去的总账。与此同时，running / waiting requests、queue interval、GPU / KV cache utilization、prefix cache hit rate 和最大稳定并发，都会告诉你系统现在是在顺滑工作，还是在被缓存和队列拖住。

> [!abstract] 定义：推理侧四个常见延迟指标
> TTFT：从请求发出到第一个 token 返回的时间。
> ITL：相邻两个输出 token 之间的延迟。
> TPOT：平均每个输出 token 的时间。
> E2E latency：一条请求从开始到结束的总时长。

RL 后训练的指标更像在看一条流水线。rollout 能不能及时供给，update 能不能及时消费，资源有没有空转，队列有没有堆起来，版本同步会不会落后，checkpoint 是否够稳，这些问题共同决定系统能否持续学习。今天很多 RLHF 系统已经把这件事说得很明白：关键不只是 loss function 写得对不对，更是 rollout、trainer、reward、评估和恢复这几段能不能接得顺。

> [!abstract] 定义：rollout、update、hybrid engine
> rollout：用当前策略生成样本或轨迹。
> update：拿这些样本做一次或多次参数更新。
> hybrid engine：让采样和学习更灵活地共享资源，尽量减少双方互相等待。

## 这章要反复纠正的三个误解

第一，`GPU utilization` 只能当线索，不能直接当结论。训练里真正要先看的通常是 MFU 和 step time；服务里更有解释力的常常是 TTFT、ITL 和并发下的退化曲线；RL 后训练里更关键的是 rollout 和 update 有没有真正重叠。卡很忙，不一定代表系统很健康。

第二，显存要看版图，不要只看总数。参数、激活、优化器状态、KV cache 都在争抢同一块 HBM，但它们出现的时刻、停留的时间和回收方式并不一样。只说“这张卡有多少显存”帮助不大，更重要的是：这块显存在什么时间被谁占着，占了多久，能不能切开、转移或回收。

第三，性能要看全链路。更快的 kernel 很重要，可它只是系统的一段。队列、prefill / decode 切分、cache hit、跨节点 KV 传输、checkpoint collective、版本同步、失败恢复，都会改写最终 wall-clock。很多“模型很慢”的问题，最后都要靠重排数据流、调度和缓存路径来解决；单点 kernel 强化往往不够。

## 让有用 token 连续地流过系统

把前面的内容压成一句话，AI Infra 的核心就是：在显存、带宽、延迟和可靠性约束下，让有用 token 连续地流过系统。

训练里，我们用 MFU、step time 和 checkpoint 成本去看这条流顺不顺；推理里，我们看 TTFT、ITL、cache hit rate 和 SLA 下的最大稳定并发；RL 后训练里，我们看 rollout 与学习能不能重叠、资源会不会空转、系统能不能稳稳恢复。场景不同，仪表盘不同，底下问的仍然是同一件事：状态对象怎样被放置、搬运、复用、回收和调度。

这一章的任务，就是把读者的注意力从“框架名字很多”拉回到“状态对象怎样流动”。只要这个视角立住，后面的训练并行、KV cache、checkpoint、serving 和 RL 数据流就会连成一条线。大模型系统的难点，不在于 GPU 数量多，而在于状态对象的生命周期很长、很拥挤、又彼此依赖。把这件事看清楚，AI Infra 就不再是一堆零散技巧，而会变成一门能反复解释现实系统的学问。
